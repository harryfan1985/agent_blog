AI 写了 30 万次提交，留下了 11 万个未解决的 bug——AI 编码的技术债真相

July 8, 2026

AI 写了 30 万次提交，留下了 11 万个未解决的 bug

Google 说 AI 写了超过 25% 的新代码。Microsoft 说是 30%。GitHub 统计了 110 万个使用 AI 编码工具的公共仓库。

但新加坡管理大学（SMU）的一个团队问了一个更实际的问题：这些代码进入生产仓库之后发生了什么？是很快被修掉了，还是像灰尘一样悄悄堆积成了技术债？

答案很残酷：24.2% 的 AI 引入 issue 最终留在了代码库里，安全问题更是高达 41.1%。而 AI 在某些方面确实在"帮忙"——它修复的 code smell 比引入的还多。但对 bugs 和安全漏洞，它是净负面的。

30 万次 AI 提交的来龙去脉

论文团队没有用分类器猜测哪些代码是 AI 写的——他们直接用 Git 元数据中的显式信号：commit message 里的 Co-authored-by trailer、author email（如 noreply@anthropic.com）、actor login（如 copilot-swe-agent[bot]）、author name（如 Cursor Agent）。

从 GitHub Archive 的 PushEvent 数据（2024.01–2025.10）中扫描，加上 API 补充查询，最终得到：

仅限 ≥100 stars、生产源文件（Python/JS/TS）、排除测试/文档/配置/第三方代码。

然后对每个 AI 提交做差分静态分析：在 parent commit 和提交本身分别跑 Pylint + Bandit（Python）或 ESLint + njsscan（JS/TS），精确归属哪些 issue 是 AI 引入的、哪些是修复的。再追踪每个引入 issue 到仓库最新 HEAD，判断是否存活。

结论很清楚：技术债不是一个工具的问题，而是 AI 辅助开发模式的系统性问题。

核心发现三：AI 修复的 code smell 比引入的还多——但对 bug 和安全是净负面

这是论文最反直觉的发现。研究者对比了 AI 提交引入的 issue vs 修复的 issue：

类型AI 引入AI 修复净变化
Code Smells431,850449,984-18,134 ✓
Runtime Bugs28,149~21,000+~7,000 ✗
Security Issues24,607~13,000+~11,000 ✗

AI 在 code smells 上是净正面的——格式问题、命名问题、未使用变量这种重复模式的清理，AI 确实能胜任。但一旦涉及需要理解程序逻辑、执行上下文、安全含义的 bugs 和 security issues，AI 引入的远比修复的多。

这是一个能力边界的精确刻画：AI 擅长 pattern matching，不擅长 deep reasoning。

旧 issue 确实在减少（>9 个月降到 19.2%），但减得太慢。累计存活 issue 从 2025 年初的几百个飙升到 2026 年 2 月的 11 万个——随着 AI 编码工具的加速采用，技术债在以惊人速度堆积。

真实案例：从 42 天才修的 bug 到至今未修的安全漏洞

  • Firecrawl（98K stars）：Devin 引入 cache=cache 参数但从未定义 cache → NameError，42 天后才修复
  • Microsoft data-formulator：Copilot 将用户输入的 table name 直接拼接到 SQL 查询 → 潜在注入，数周后才重构
  • CrewAI brave_search_tool.py：Devin 引入 requests.get() 无 timeout — 可能导致永久阻塞，至今未修复
  • Intel librealsense（8.6K stars）：Copilot 引用了未定义的常量 → 运行时错误，三周后修复
  • ArchiveBox（27K stars）：Claude Code 引入 bare except: pass + 不指定文件编码 → 几小时内修复（正面例子）

对 Coding Agent 的工程启示

1. 权限分级：safe ops vs dangerous ops

数据明确显示 AI 在安全问题上净负面。Coding agent 的自动提交权限应按操作类型分级：

  • Safe（允许自动提交）：格式化、命名修正、未使用变量清理、简单重构
  • Dangerous（标记为需要人工 review）：subprocess 调用、SQL 拼接、网络请求、认证逻辑、加密相关

这不是要否定 agent 的能力，而是承认它的真实能力边界——paper 的数据提供了这个边界的实证基础。

2. Agent 的 "code review" 不可靠，尤其是正确性和安全性

AI 在 code smells 上修复多于引入，但在 bugs 和 security 上相反。这说明一个 agent 可以很好地审查代码风格，但无法可靠地审查代码正确性和安全性。

如果 agent pipeline 中有 "自我审查" 环节，这个环节不应该由 LLM 单独承担——至少需要静态分析工具（Pylint + Bandit + ESLint）作为 hard gate。

3. 量化 Agent 的 "代码可持续性"

论文的 debt lifecycle tracking 方法论可以直接转化为 agent 的 KPI：不是衡量 "agent 生成了多少代码"，而是衡量 "agent 引入的 issue 在多长时间内被修复"。一个 agent 如果 24.2% 的 issue 存活超过 9 个月，那么它的 "生产力贡献" 需要打折扣。

局限与诚实标注

  • 无人类对照：论文没有纯人类提交的 baseline。论文自己承认：真实仓库中人机混合太严重，无法构建干净对照。所以不能回答 "AI 写的代码比人写的差多少"——只能回答 "AI 引入的 issue 后来怎么样了"。
  • 静态分析的误报：Pylint/Bandit/ESLint 都有 false positive。论文未做误报率评估。
  • 存活追踪的低估风险：如果文件被删除或完全重写，issue 被标记为 "已解决"——可能低估真实存活率。
  • 范围限制：仅 Python/JS/TS、≥100 stars 仓库、公开仓库。私有仓库和小项目的情况未知。

一句话

AI 写代码的"生产力红利"不是免费的——每 4 个 AI 引入的质量问题就有 1 个留在代码库里成为长期技术债，安全问题存活率更高达 41%。AI 在 pattern 类清理上确实在帮忙，但在需要深层理解的安全和正确性问题上，它引入的远比修复的多。
