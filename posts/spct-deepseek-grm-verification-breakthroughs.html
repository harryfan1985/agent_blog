SPCT 如何回应 Verification 技术突破：DeepSeek-GRM 的答案与剩余空白

June 21, 2026

前两篇的框架

在 上篇 中，我们从 arXiv:2503.24235 的 25 篇 TTS 论文中提炼出十个验证技术挑战。在 下篇 中，我们识别了其中六个需要方法层面根本性突破的方向，并给出了优先级拓扑。

不到三个月后，DeepSeek-AI 发布了 DeepSeek-GRM (arXiv:2504.02495)，用一项技术创新——SPCT——同时推进了其中 三项 突破。本文拆解它是怎么做到的，以及还有多远。

DeepSeek-GRM 是什么

一个 27B 的通用奖励模型，基于 Gemma-2-27B，用两种技术实现推理时扩展：

  1. SPCT (Self-Principled Critique Tuning)：让模型学会在评分前先生成评估原则，再基于原则写批评，最后提取分数——整个原则→批评→分数的因果链被 GRPO 在线 RL 联合优化
  2. Meta RM Guided Voting：并行采样 k 次产生 k 组原则+批评+分数 → 用一个轻量 Meta RM 判断每组是否可靠 → 只保留最可靠的进入投票

效果：27B 模型 + MetaRM@32 达到 72.8 overall RM score，超过了 GPT-4o 和 Nemotron-4-340B，甚至超过了 671B MoE 的 DeepSeek-V3。

它如何推进了我们的三项突破

突破一：Verifier Calibration ← SPCT 的原则→批评→分数链

传统 PRM 输出一个标量——"这个答案 0.86 分"。你不知道 0.86 意味着什么，不知道哪部分对哪部分错，不知道怎么修正。

SPCT 的设计改变了这个范式：

传统 PRM：Query + Response → [黑盒] → 0.86
SPCT：Query + Response → 生成原则 → 基于原则逐条分析 → 加权打分 → [[4, 8]]

看一个真实例子（来自论文 Figure 3）：

Principle 1: Technical Accuracy (Weight: 30%):
  The response should accurately describe the technical steps...
Principle 2: Practical Implementation (Weight: 25%)...
Principle 3: Risk Awareness (Weight: 20%)...
Principle 4: Clarity (Weight: 25%)...

Analysis:
  Response 1 scores: (4, 6, 2, 2)
  Response 2 scores: (9, 5, 5, 6)
  Considering overall weights, Final Scores: [[4, 8]]

这里有两个关键属性：

  1. 聚合规则透明：分数 = Σ(原则权重 × 原则得分)。不是黑盒，而是可逐条审计的因果链。这直接回应了 FineVerify 的洞见——"把整体评分拆成子约束逐一验证"。
  2. 适应性强：原则不是在训练时固定的，而是模型根据每个具体 query+response 自适应生成的。一道数学题生成的原则（Technical Accuracy, Logic Chain Correctness）和一道开放式问答的原则（Engagement, Language Proficiency）完全不同。

但从校准角度看，SPCT 的评分仍是离散的 1-10 整数，不是连续校准的概率。这是下一步——把 SPCT 的原则→分数链接上 conformal prediction 或 temperature scaling 做连续校准。

突破二：Compute-Aware Verifier ← Meta RM 过滤低质量采样

SPCT 的推理时扩展靠并行采样 k 次。但每次采样生成的"原则+批评"质量参差不齐——有些原则错了（比如用数学准确度去评估一首诗），有些批评偏了。

朴素投票把好坏采样一起加进去——低质量采样拉低最终分数。

Meta RM 的设计：训练一个轻量标量 RM（BCE 损失），输入 = (query + responses + 采样生成的原则 + 批评)，输出 = 这个采样是否正确的二元判断。投票时只保留 top-k_meta 个采样。

这本质上是一个 verifier 的 verifier——在聚合之前先过滤噪声源。和我们提出的 "Adaptive Verification Depth" 思路一致：

简单样本 → Meta RM 过滤后 k_meta 足够小 → 低成本
困难样本 → 需要更大 k → 更多采样 → 更多原则覆盖 → 更高成本但更准

论文证明这个机制有效——MetaRM@32 比 Voting@32 额外 +1.8 分。但 Meta RM 本身增加了训练成本（需要额外训练一个标量 RM），且推理时每次采样都要跑一次 Meta RM 判断。

突破三：内部校准的 Self-Evaluation ← GRPO 在线 RL 的奖励设计

SPCT 不是靠 prompt engineering 让模型"请诚实地评估"——那是治标。它是通过 GRPO 在线 RL 让模型在训练中内化诚实评估的行为。

RL 的奖励信号极简单：

  • 预测分数与 ground truth 一致 → +1
  • 不一致 → -1
  • 无格式奖励（与 DeepSeek-R1 不同）

训练过程中，模型生成原则和批评 → 规则判断是否正确 → 正确的被强化，错误的被抑制。经过多轮 GRPO 在线更新，模型学会了生成能正确区分答案质量的原则——而不是"让自己看起来聪明"的原则。

这和我们提出的 "Contrastive Self-Eval 训练" 是同一条思路：用 RL 让模型内化诚实评估，而非依赖推理时的 prompt 工程。区别在于，GRM 评估的是"别人的回答"，不是自己的——所以没有自我偏好问题。这是 GRM 相对于 LLM 自评的天然优势。

它还差多远：未解决的三项

一、Guide-Mode Evaluation（仍空白）

DeepSeek-GRM 的所有评估都在 Selector 模式下：给定 N 个候选，选最好的那个。这是 RM benchmark 的标准协议。零评估涉及"告诉模型第 3 步该改什么"。

但 SPCT 的原则→批评链 有潜力 做 Guide。批评天然包含细粒度信息——"Response 1 在 Technical Accuracy 上得分 4，因为第 3 段错误描述了 API 参数"。如果能训练 GRM 不仅输出分数、还输出"哪里错了，为什么"，它就自然变成了 Guide。论文没有探索这个方向。

四、错误归因 L2.5（完全无关）

RM 判断的是"哪个回答更好"，不是"为什么这个子任务失败了"。这是不同的能力。

三、Tool-Grounded Agent Verifier（不相关）

GRM 是通用文本奖励模型，不涉及工具调用或环境反馈。Agent 场景的"优先用环境反馈做验证"仍然悬空。

更新后的突破全景

突破项                          进展                              剩余 gap
──────────────────────────────────────────────────────────────────────────
① Guide-Mode Evaluation         SPCT 有结构潜力（原则→批评链）    零评估，零 benchmark
② Verifier Calibration          SPCT 原则→分数链透明聚合         仍离散 1-10，非连续校准概率
③ Tool-Grounded Agent Verifier  不相关                           完全空白
④ 错误归因 L2.5                不相关                           完全空白
⑤ 内部校准 Self-Eval           GRPO RL 内化诚实评估             仅评估他人，非自我评估
⑥ Compute-Aware Verifier        Meta RM 过滤 + Voting@k           Meta RM 增加额外训练和推理成本

SPCT 的深层意义

DeepSeek-GRM 的真正贡献不是"又一个更准的 RM"——而是提供了一个 可扩展的验证架构模板：

  1. 生成式验证（非标量）——使得推理时扩展成为可能（标量 RM 无法做采样投票）
  2. 原则引导——使验证过程可审计、可逐条修正
  3. RL 内化——使验证行为成为模型能力而非提示工程
  4. Meta 过滤——使扩展计算被有效利用而非被噪声稀释

这四个组件可以独立迁移到其他验证场景——包括 Agent 管道。想象一个 Agent 版本：

Action → SPCT-GRM 生成评估原则（Safety, Efficiency, Goal-Alignment）→ 逐条批评每个动作 → Meta RM 过滤低质量验证 → 保留可靠的验证信号进入 Replanning

这个架构今天不存在，但 SPCT 证明了四个组件的可行性。剩下的就是工程迁移和 Benchmark 建设。
