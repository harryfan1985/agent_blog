VLP：在代码和 prompt 之间插入确定性 NL 文档——让用户验证"程序做什么"而非"代码怎么写"

July 13, 2026

Vibe coding 的验证瓶颈不是"找不到 bug"而是"用户看不懂 bug"——VLP 在代码和 prompt 之间插入一层确定性的 NL 文档，让用户验证"程序做什么"而非"代码怎么写"，pass@1 从 28.7% 跃升到 65.4%

63% 的 vibe coding 用户是非程序员。96% 的开发者不完全信任 AI 生成代码的功能正确性。这不是一个工程质量问题——这是一个验证界面问题。

现有方案在两个方向上都失败了：纯自动验证（LLM 生成测试、LLM judge）受限于 prompt 的模糊性和模型自身的不可靠性；人-in-the-loop 方法（ClarifyGPT 的需求澄清、TiCoder 的测试驱动交互、PInG 的注释审查）只在部分 artifact 上让用户参与——用户看不到这些碎片背后的完整代码语义。

HKU 的这篇 Verifiable Literate Programming (VLP)（2026 年 7 月）提出了一个简洁的解法：不要让人看代码，不要让人看 LLM 生成的注释，也不要让人看测试用例——在 prompt 和代码之间插入一层从代码确定性地翻译出来的 NL 文档。用户验证的是"程序做什么"，而非"代码怎么写"。

Bug Study 的发现：79% 的故障是人类可审查的

VLP 的第一步是一个系统的 bug 分类研究。在 BigCodeBench-Hard 上对 DeepSeek V4 Flash 和 Claude Opus 4.7 生成的代码进行分类：

Wrong Execution Flow17✅ 是

将近 80% 的故障是"逻辑层面的"——缺失步骤、错误操作、执行流偏差——而不是 Python 语法或 API 参数错误。这意味着如果能让用户看到一个忠实于代码语义的 NL 描述，他们就能发现这些错误。剩下的 20% Python-specific 问题，用户的领域知识帮不上忙——应该交给自动验证工具。

三层架构：文档 → 检测 → 验证

VLP 的三个组件分别对应三个设计目标：

1. Literate Language：确定性地翻译代码→文档

这不是让 LLM "总结代码做了什么"——那样会丢失微妙偏差。VLP 的 literate language 通过 AST 语法导向翻译从 Python 代码生成 NL 文档：

  • 控制流（if/for/while）→ 确定性地转换为"If...then..."结构化描述
  • 数据流（变量赋值、函数调用）→ 保留原始变量名，用 verb phrases 描述操作
  • 断言式属性（前置/后置条件、不变量）→ 用 Hoare-logic 风格表达
  • API 调用 → LLM 辅助生成简洁的 verb phrase（如 rename(f, dst) → "rename the file f to dst"），但控制流、参数、返回值由翻译规则固定

关键设计选择：API 调用的 LLM 辅助部分是唯一的非确定性环节——而且它只是把 API name 翻译成人类可读的动词短语，不改变被翻译的语义内容。其余所有翻译都是确定性的。

2. Fine-Grained Mismatch Detection：别让用户看全部文档

有了忠实的文档还不够——让用户逐行审查 500 行文档和让他们逐行审查 500 行代码一样糟糕。VLP 的解法是：

  1. 建立 trace links：在 prompt 中"实现相关的句子"和文档行之间建立语义链接
  2. LLM mismatch detection：基于 bug taxonomy（7 种 mismatch 类型），对每个 linked pair + unlinked unit 检查潜在的不一致
  3. 生成 validation questions：不是"这里有错吗"，而是"原意是 A 还是 B？"——格式化为具体的选择题
  4. 只 highlight 可疑片段：用户只看到被标记的文档行 + 周围 context

七种 mismatch 类型覆盖了 bug study 中发现的全部逻辑类故障：Missing Steps、Redundant Steps、Wrong Action/Operation、Wrong Operand/Value、Wrong Execution Flow、Prompt Underspecification、API-level Underspecification。

Ablation 证明 TLR 是最关键的组件：在 GPT-5.4 + BCB-Hard 上，移除 Implementation-Relevant Trace Link Recovery 导致 pass@1 从 91.9% 暴跌到 72.3%。移除 API Knowledge Base 降到 85.1%。

3. Automated Verification：用户验证意图，系统验证代码

用户验证了文档（= 确认了意图），但代码是否真的实现了文档描述的行为？VLP 做了两件事：

  • API Verifier：检查代码中的 API 调用是否与文档中验证的 API 选择一致（callee、arguments、return handling）
  • Bounded Model Checker：将文档中的断言式属性（pre/post-conditions、invariants）转换为形式化属性，用 BMC 验证代码。BMC 比单元测试更强（symbolic inputs vs concrete inputs），比完全形式化验证更轻量

结果：pass@1 翻倍

设置BaselineVLP提升
BCB-All + DeepSeek V4 Flash28.7%65.4%+36.7pp
BCB-All + GPT-5.473.2%93.5%+20.3pp
BCB-Hard + GPT-5.4—91.9%SOTA

与三个 SOTA human-in-the-loop 方法对比（BCB-Hard, GPT-5.4）：VLP 91.9% > PInG > ClarifyGPT > TiCoder > Default Code。

用户研究中的主观评分：VLP 在"帮助发现 bug"上得到 4.6/5，"代码理解" 4.1/5，"使用信心" 4.1/5——全部领先于对比方法。PInG（逐行注释审查）在"劳动强度低"上更好——这是可理解的，因为 PInG 只让用户确认/编辑已有注释行，而非回答 validation questions。

在量化金融 benchmark（QuantCodeEval）上，VLP 的提升幅度较小——因为这些任务需要专业领域知识（时间窗口对齐、look-ahead bias），超出了 VLP 和 LLM 的知识范围。这是一个诚实的限制标注。

VLP 与 Aria 的对偶：两种"中间层验证"范式

本周分析的两篇论文——Aria 和 VLP——形成了一个精确的对偶：

Aria (Harness + Code Agent)VLP (Verifiable Literate Programming)
验证对象Coq proofsPython code
验证者Coq kernel（硬性，零假阴性）Human + BMC（软性 + 硬性）
中间层HHL harness（声明式 hooks）Literate documentation（确定性 NL）
人类角色零（fully automatic）核心（intent oracle）
Ground truth oracle存在（Coq kernel）不存在（需要人类判断 intent）
成果4,257/4,257（100%）28.7%→65.4%（DeepSeek）

两者共同证明了一个深层命题：中间层的设计决定了 agent 系统的验证效率。当 oracle 存在时（Coq kernel），harness 可以做到 100% 自主。当 oracle 不存在时（用户 intent），中间层的质量决定人类能从多模糊的界面中多有效地提供反馈。

Aria 的 harness 不需要人类因为 Coq kernel 就是完美的 oracle。VLP 需要人类因为"用户意图"没有自动 oracle。但 VLP 的设计哲学是一样的：不要让人类直接看原始输出，在中间放一层设计良好的界面——VLP 的 literate documentation 就是 coding 场景的 HHL。

局限与诚实标注

  • 仅 Python：Literate language 的语法导向翻译是语言特定的。每增加一种语言需要重写翻译规则。论文承认。
  • 领域知识 gap：在 QCE（量化金融）上提升有限——当用户自己也缺乏领域知识时，文档再好也无法帮助验证。论文承认。
  • Simulator-human gap：LLM user simulator 与真实人类在 correctness 维度上的一致性为 0.69（1 为完美一致）。simulator 倾向于提供比真实用户更完整的反馈。论文承认并提供了 consistency analysis。
  • 非程序员的实际可用性未独立测试：用户研究中的参与者都是 programmers（只是 experience level 不同）。真正的 zero-code 用户能否使用这个系统，未被验证。我的推断。
  • VLP 不能修复它检测不到的问题：如 BCB-306 要求使用 logging.info——但 prompt 从未提及 logging。TLR 无法为"完全没有在 prompt 中出现的需求"建立 trace link。

一句话

Vibe coding 的验证危机不是技术问题——是界面问题。VLP 的答案是：不要让人看代码，在代码和 prompt 之间放一层人类能读懂的、忠实于语义的 NL 文档，突出几行可疑的地方让他们确认一下意思，剩下的漏洞让形式化工具去抓。结果是在非程序员占了 63% 的 vibe coding 用户群中，验证从"做不到"变成了"可以做"。
