AfterVibe：Vibe Coding 结束后，留下 Spec 而非代码

July 13, 2026

一句话：vibe coding 结束后，留下的不应该是代码——而是一份可以让另一个 AI 盲写等价的 spec

Meta 的研究人员提出了 AfterVibe：从 vibe coding 会话中自动提取自然语言规格说明书（spec），并通过 regeneration test 验证——让一个"盲" agent 只读 spec 重新实现，然后与原始代码做三层验证比对。在 72 个公司内部真实 vibe coding 项目上，AfterVibe 的 spec 达到平均 5.06/6.0 的再生分数，迭代强化后升至 5.74，显著优于开发者手写的代码审查描述（4.23）。论文的核心主张是：在 AI 生成代码量超过人工审查能力的时代，spec 应该取代代码成为人类审查的主要产物和记录源头。

核心机制：Retrospective Spec Recovery + Regeneration Test

AfterVibe 的输入很简单：一次 vibe coding 会话的对话轨迹 T（开发者与 AI agent 的消息）和最终代码变更 C（unified diff）。输出是一份结构化的自然语言 spec。

工作流程五步走：

步骤做什么关键设计
➀ Spec DistillationLLM 读 T + C，提取两部分：
Part A：意图与理由 + 关键设计决策 + 不可发现事实
Part B：行为需求清单（declarative statements）		提示词强制抽象——"不要写代码、标识符、路径、行号，除非是不可发现的事实"
➁ VC Extraction从 T 和人工 summary 中提取验证条件（verification conditions）不依赖 spec——保证 VC 是独立 oracle
➂ Test Extraction从会话提取可执行测试命令先验证在 C 上能通过，确保 oracle 有效
➃ Blind Regeneration一个独立 agent 只读 spec（无会话上下文），在仓库基础上重新实现独立会话，防止信息泄漏
➄ Three-Tier Grading三层验证：flex test ×3 + VC check ×2 + ground-truth alignment ×1regen_score = 3×flex + 2×vc + align（0–6）

三层验证的独立性是关键。VC 不是从 spec 导出（否则循环验证），而是从对话轨迹独立提取；flex test 允许接口层面的微小变化；GT alignment 由 LLM agent 判断再生代码是否在语义上符合对话中表达的意图。三者互补：VC 检查离散行为属性，flex test 检查可执行正确性，GT alignment 检查整体语义一致性。

关键发现一：平均再生分数 5.06/6.0——spec 确实能驱动盲写

在 72 个通过了 ground-truth 验证的真实项目上（来自 113 个原始候选，排除基础设施失败和 oracle 不通过的任务）：

指标数值
平均再生分数5.06 / 6.0
VC 通过率86.8%
Flex test 通过率72.1%
GT alignment 通过率66.2%
原始 diff 平均大小41,638 字符
Spec 平均压缩比5.69×（约 6,960 字符）

对比基线：

方法再生分数VC%Flex%GT%
开发者手写 summary（用作 spec）4.2364.060.050.0
AfterVibe5.0686.872.166.2
AfterVibe + strengthening5.74–––

+0.82 分的差距是实质性的——开发者手写的代码审查描述本质上是写给有代码在旁的人类审阅者看的，它省略了大量行为约束和不可发现事实，而这些恰恰是盲 regen agent 需要的。

关键发现二：迭代强化——84% 的弱 spec 任务可被修复

AfterVibe 的 strengthening loop 按单个 VC 粒度进行反馈驱动优化：

  • 对每个失败的 VC，refinement LLM 收到原始 spec + GT diff + regen diff + 失败原因
  • 分析 spec 中哪里模糊或缺失 → 生成完整改写版 spec S′
  • 重新跑完整管道（新 agent 盲写 → 三层验证）
  • Best-of-K 保留每任务最高分

结果：基线 31/68 任务未达满分 → 3 轮后，26/31（84%）实现了正向提升，缺省队列的累积分数从 3.93 稳步升至 5.51。

VC 通过率在 strengthening 过程中保持高位（82–89%），但 flex test 和 GT 通过率在后期轮次下降——不是方法变差，是剩余队列本身更难。

关键发现三：抽象 vs 具体的取舍——不是越具体越好

作者测试了三种提取策略的 tradeoff：

策略Spec 平均长度压缩比再生分数代码泄漏率
Constrained（~1000 字符上限）2,11716.16×4.8123.7%
Standard6,9605.69×5.0638.6%
Diff-preserving（保留具体实现）11,8362.79×5.5562.6%

表面上看，更具体的 spec 分数更高。但 leakage 分析揭露了真相：diff-preserving 策略中 62.6% 的原始 diff 代码 token 被逐字复制到 spec，其分数优势主要来自 GT alignment grader（89% vs 63–66%）——它在奖励字面匹配而非意图传达。

Standard 是最佳平衡点：5.69× 的有意义压缩 + 中等泄漏 + 强行为验证。

关键发现四：再生多样性——spec 约束"做什么"但不规定"怎么做"

每个任务跑 3 次独立再生，代码相似度分析：

  • 均通过的 pair：chrF 0.89（不是 1.0）——同一 spec 产生结构不同的实现
  • 均失败的 pair：chrF 0.86——spec 的系统性缺口引导 agent 走向相同的错误
  • 混合结果 pair：chrF 0.84——真正的分歧驱动了结果分裂
  • 上界（自己 vs 删一行）：chrF 0.98；下界（跨任务无关 pair）：chrF 0.18

跨轮次稳定性好：3 次 rollouts 的再生分数均值为 5.06/4.87/4.99，每任务标准差仅 0.49。

关键发现五：Oracle 有效性——负对照实验证明 grader 不是虚设

论文用 spec-free 的负对照实验验证了 grader 的可信度：

代码质量VC 通过率Flex test 通过率
正确代码（ground truth）100%100%
部分代码（随机删 50% hunks）47.0%42.4%
变更前代码（parent commit）0.0%58.0%
错误代码（permutation）1.4%51.4%

VC 对错误代码的区分度极强（0% 和 1.4%），而 flex test 对变更前代码仍有 58% 通过率——暗示部分测试依赖已有功能。这验证了 VC 作为 oracle 的核心价值。

Environmental Grounding Hypothesis：为什么 spec 可以这么短

论文提出了一个关键假设并提供了经验支持：coding agent 可以从仓库环境中推断大量上下文——构建配置、命名约定、模块结构、API 契约——而不需要 spec 拼写出来。

"传统上被认为属于 'spec' 的大部分内容其实是一个 grounded agent 可以从代码库中推断的共享知识。spec 真正需要显式书写的部分是项目特定的、无法从环境中推导的决策。"

这意味着：好的 spec 对可发现事实保持抽象，只对不可发现事实保持具体。这解释了为什么 5.69× 压缩比的 spec 仍能驱动功能等价的再生。

对 AI 工程的启示

1. Spec 作为代码审查的主要产物

论文明确主张：当 AI 生成代码的速度超过人工审查能力时，人类应该审查 spec 而非代码。spec 是意图的持久化表达，代码只是该意图的一次实现。这个转变对工程实践的影响深远——意味着 PR 的首要产物可能从 diff 变为 extracted spec。

2. Regeneration test 作为 spec 质量度量

传统上 spec 质量没有客观度量。AfterVibe 提供了：如果另一个盲 agent 能复现功能等价的代码，spec 就是充分完整的。这对任何使用 spec-driven development 的团队都是可操作的质量门禁。

3. Vibe coding 不需要以代码告终

论文展示了一个闭环：vibe code → extract spec → verify spec → (可选) 强化 spec。这意味着 vibe coding 可以无缝衔接到更传统的软件工程实践——spec 成为可版本化、可审查、可测试的工程产物。

局限与诚实标注

  • 数据集范围（论文自认）：72 个任务来自单一公司的内部 monorepo。语言、领域、开发者群体的泛化性未验证。
  • Oracle 质量：VC 和 flex test 均由 LLM 生成。负对照实验虽证明区分度，但生成质量对 LLM 能力有依赖。
  • Leakage 问题：standard extraction 有 38.6% 代码泄漏——spec 中仍有大量原 diff 的 token。论文坦诚此点但未量化泄漏对再生分数的因果影响。
  • 再生稳定性：每任务标准差 0.49 虽小，但意味着同一 spec 的不同 rollout 仍有不可忽略的分数差异。在生产环境中需要多次运行取 max/min。
  • Environmental grounding 的边界（我的推断）：假设在陌生或结构混乱的仓库中，agent 提取环境上下文的能力会大幅下降，spec 可能需要更长。论文未测试此情景。
  • Spec 的"隐藏假设"：spec 的意图和设计决策部分由 LLM 从对话中推断——这意味着低频术语、领域知识、非标准约定的提取质量未经验证。
  • "Spec 取代代码作为记录源头"的可行性（我的推断）：论文展示了 spec 足以驱动再生，但未展示在实际开发生命周期中（多人协作、持续集成、backward compatibility、安全补丁）这种模式的可维护性。

一句话

AfterVibe 证明了 vibe coding 对话中蕴含的开发者意图可以从对话轨迹中自动蒸馏为可验证的规格说明书——这份 spec 不仅比开发者手写的审查描述更强，而且迭代地可改进，足以让另一个盲 agent 重新实现功能等价的代码。这意味着 AI 时代的软件工程可能正在向"spec 是记录源头，代码是实现细节"的模式转变。
