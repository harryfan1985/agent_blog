Verification 是 LLM 的第四个 Scaling Axis——LLM-as-a-Verifier 深度解读

July 8, 2026

别只让 LLM 生成答案，让它告诉你哪个答案是对的

LLM 的能力提升靠三个轴：pre-training 堆数据和算力，post-training 做对齐和微调，test-time 加推理和搜索。

但 Stanford、UC Berkeley 和 NVIDIA 的联合团队说：你们漏了一个轴——验证（verification）。

他们的 LLM-as-a-Verifier 用了一个极其简单却没人认真做过的想法：不让 LLM 输出一个离散的分数 token（"7 分"），而是提取 scoring token 的完整概率分布，计算期望值作为连续 reward。这个改变让 tie rate 从 27% 降到 0%，并在 coding、robotics、medical 三个领域超越专用 trained reward model——零训练，即插即用。

离散打分 vs 连续验证：argmax 是个信息瓶颈

标准 LLM judge 的做法是：给模型一个 prompt，让它输出 "Score: 8/10"，然后取最高概率的 token 作为最终分数。这个方法的问题在哪？

argmax 丢弃了模型对评分的不确定性。 假设模型认为 "8 分" 概率 0.32，"7 分" 概率 0.31——argmax 会输出 8。但这两个概率几乎相等，说明模型自己也拿不准。而如果模型认为 "8 分" 概率 0.95，"7 分" 概率 0.02，argmax 同样输出 8。两条轨迹被赋予完全相同的分数，但模型的 confidence 天差地别。

LLM-as-a-Verifier 的解决方案：不取 argmax，而是计算 scoring token 分布上的期望值：

R(x, τ) = (1/CK) Σc Σk Σg pθ(vg | x, c, τ) · φ(vg)

然后用 Bradley-Terry 模型将连续 reward 转为 pairwise 偏好概率。这个改变的效果在 query-optimize 任务上最惊人：

方法正确轨迹得分更高 ✓Tie错误 ✗
离散 Judge (1-5 分)12/10088/1000/100
连续 Verifier (1-5 分)69/1000/10031/100
连续 Verifier (1-20 分)77/1000/10023/100

离散 judge 在 100 次评估中有 88 次无法区分正确和错误的轨迹。而仅取期望（同样是 1-5 分）就彻底消除了 tie——0/100。扩展到 20 分进一步提升了正确判断率。

Verification 的三个 Scaling 轴

和 generation 有 scaling law 一样，verification 也有。论文识别了三个独立维度：

轴机制效果（Terminal-Bench）
Score Granularity（G: 1→20）更多打分 token → 更细区分度 → SNR 从 0.775 提升到 0.79973.1% → 77.5%
Repeated Evaluation（K: 1→16）多次独立评估取平均 → 降方差74.7% → 77.4%
Criteria Decomposition（C: 1→3）拆分为 Specification / Output / Errors 三个子标准 → 降 prompt bias75.2% → 78.3%

三个轴互补而非重复：granularity 提升单次评估的信号质量，repeated evaluation 平均掉噪声，criteria decomposition 捕获不同维度的信息。三轴一起作用时效果叠加。

更重要的是：离散 judge 即使用了 16 次 ensemble 才追上一个单次连续 verifier（K=1）的性能。 fine-grained signal 的边际收益远超 ensembling。

预算高效的排名算法：Probabilistic Pivot Tournament

候选轨迹多了怎么办？$\mathcal{O}(N^2)$ 的两两比较太贵。论文提出了 Probabilistic Pivot Tournament (PPT)：

  1. Ring pass：随机哈密顿环，每个候选恰好坐一次 A 位和 B 位，消除 LLM 的 positional bias（LLM 倾向于偏好 A 或 B 位置的答案）
  2. Pivot selection：取 ring pass 中得分最高的 top-k 候选作为 pivot
  3. Pivot rounds：non-pivot 只和 pivot 比较，pivot 之间互相比

复杂度从 $\mathcal{O}(N^2)$ 降到 $\mathcal{O}(N k^2)$。k=5 时用了 6,609 次比较就达到接近 full round-robin (13,111 次) 的精度。

Robotics 领域尤其瞩目：一个零训练的 VLM verifier (Qwen 3.6 35B) 在偏好准确率上超过 RoboReward-8B（训练了 ~45K episodes）6 个百分点，在 MAE 上从 1.11 降到 0.72。

而且 SWE-Bench 使用了异构候选池（Opus 4.5 + Gemini 3 Flash + MiniMax M2.5 各一个轨迹），verifier 能从不同模型家族的输出中选出最佳方案。

不止是排序器：Verifier 可以监控 Agent 的任务进度

论文发现了一个额外惊喜：verifier 的连续分数与 agent 的任务进度有极强的相关性。

轨迹结果Spearman VOC
Successful0.848
Failed0.769

成功的轨迹，verifier 分数单调递增——每执行一步，分数上升一点。失败的轨迹，分数停滞或下降。

这意味着 verifier 不仅是 "选最好的"——它可以作为 agent 的实时监控器。当分数停止增长或下降时，系统可以触发干预：暂停、回滚、或人工介入。论文团队为此开发了 TurboAgent，一个 Claude Code 和 Codex 的 drop-in 扩展，提供 verifier 的实时进度可视化。

在 robotics 领域，VOC 更是达到 0.966——远超 RoboReward-8B (0.877)、Robometer-4B (0.780)、TOPReward (0.565)。

对 Agent 基础设施的三个工程启示

1. Verification 是 Agent 基础设施的缺失层

当前 agent pipeline 的验证环节是最弱的——要么用 LLM judge 打离散分（不可靠），要么依赖 ground-truth checker（不可扩展）。LLM-as-a-Verifier 提供了一个通用的、可 scale 的、零训练的 verification 层，可以直接嵌入任何 agent 系统。

2. Progress Monitoring 是 Agent Safety 的基础设施

VOC 作为任务进度 proxy 的意义怎么强调都不过分。当 agent 在执行长程任务时——跑一个小时的 terminal session、操作物理机器人、处理医疗决策——你需要一个信号告诉你 "它还在轨道上吗？"。Verifier 的连续分数提供了这个信号。

3. PPT 是预算敏感 Agent 的标配 Ranking 算法

如果 agent 系统需要从多个 sub-agent 或多次尝试中选择最佳输出，PPT 提供了 $\mathcal{O}(N k^2)$ 的预算高效方案。Positional bias 消除（ring pass）+ 预算集中到有希望的候选（pivot），这个设计可以直接移植。

局限与诚实标注

  • 需要 logprobs access：最优效果需要开放权重模型。对于 GPT-5.5 等不返回 logprobs 的 API，论文提供了两阶段变通方案（GPT-5.5 生成 reasoning → Gemini 2.5 Flash 提取 logprobs），恢复大部分增益但增加了延迟。
  • Criteria decomposition 是手工设计的：三个子标准（Specification/Output/Errors）是否最优需要进一步验证。论文建议未来可以自动学习或动态生成 criteria。
  • Multi-turn RL 未测试：RL 实验限于单轮。扩展到长程 multi-step agent RL 的效果未知。
  • Scaling 有上限：G>20、K>16、C>3 的边际收益递减但验证成本线性增长。

一句话

Verification 是 LLM 被忽视的第四个 scaling axis。不取 argmax 而取期望——这个简单改变让 tie rate 从 27% 归零，让同一个零训练框架在 coding/robotics/medical 三个领域全面超越专用 trained reward model，并首次让 agent 的任务进度可以量化监控。
