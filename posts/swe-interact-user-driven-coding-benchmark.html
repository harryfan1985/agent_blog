解决率腰斩：当 SWE Benchmark 从自主变成交互式开发

July 3, 2026

核心发现：Scale AI 团队将 SWE benchmark 从单次自主实现重构为多轮用户驱动的交互式开发流程。结果令人警醒——最强的模型在单轮任务上能达到 ~50% 的解决率，但在同样的任务换成交互式设定后，解决率直接腰斩到 ~25%。这不是"模型不够聪明"的问题，而是暴露了一个当前 benchmark 完全未覆盖的能力维度：在模糊指令下发现用户意图、在增量反馈中迭代、在需求演化中保持正确性。

一、SWE Benchmark 的盲区

现有的 SWE benchmark（SWE-bench、SWE-bench Pro、SWE Atlas、DeepSWE）有一个共同假设：任务需求在开始时就是完整给出的。Agent 拿到一份详细的问题描述，然后自主实现，最后跑测试验证。

但真实的开发场景完全不是这样。Scale AI 团队分析了 SWE-chat 数据集中的数千条真实用户-Agent 交互记录，发现：

  • 用户的初始指令通常是 简短、不完整、模糊 的
  • 开发过程是一个迭代循环：GOAL DISCOVERY → PLAN → APPROVAL → IMPLEMENT → REVISE → SUBMIT
  • 最常见交互模式是 vibecoding（Agent 写 >99% 代码，用户不手动写代码）
  • 最常见用户画像是 Expert Nitpicker——消息简短、直接、迭代式 critique、关注 API 细节、逐步追加需求

这意味着：现有的 SWE benchmark 测量的是 自主实现能力，而非真实开发中最核心的 交互式目标发现与迭代精炼能力。

二、SWE-Interact：把 Benchmark 从自主变成交互

SWE-Interact 的架构设计有三个关键组件：

2.1 用户模拟器

这不是一个简单的"Agent 问了就给答案"的 API 调用。用户模拟器是一个 有工具、有 persona、能检查 Agent 工作区的 agentic 实体：

  • Persona 设计：基于 SWE-chat 真实数据建模的 Expert Nitpicker——消息简短、挑剔 API 细节、逐步追加需求、会检查 Agent 的代码后再给反馈
  • 工具访问：用户模拟器有 shell 访问权限，可以在 Agent 的 workspace 中运行 git、grep、sed、find 来检查实现——这是 real user 的行为，不是静态的 API 调用
  • 渐进式披露：不会一次性给出所有需求，而是从模糊描述开始，检查 Agent 的 plan 和实现后，逐一追加要求

2.2 任务构建

从三个前沿 SWE benchmark 中选取 25 个任务各一（共 75 个），选择标准是"指令复杂、需求分层、天然支持多轮"。关键设计：

  • 同一任务的单轮版本和多轮版本使用 相同的 verifier
  • 这使得可以精确测量："同一个实现任务，从自主变成交互后，到底难了多少"

2.3 执行基础设施

基于 Harbor 框架 + Modal 容器化沙箱。Agent 通过 MCP 托管的 ask_user 工具与用户模拟器交互。

三、核心结果：解决率腰斩

Model单轮 Resolve Rate多轮 Resolve RateΔSteps (多轮/单轮)Cost (多轮/单轮)
Opus 4.850.7%26.7%-24.0pp3.2X2.3X
GPT 5.548.0%24.7%-23.3pp3.9X3.5X
Gemini 3.5 Flash29.3%17.3%-12.0pp3.0X2.9X
Kimi K2.625.3%14.7%-10.7pp1.7X4.5X
Sonnet 4.621.3%18.8%-2.5pp3.3X2.1X

两个核心发现：

  1. 单轮性能 ≠ 多轮性能：单轮能力最强的模型（Opus 4.8, GPT 5.5），在多轮设定下也降幅最大（-24pp）。强自主 ≠ 强交互。
  2. 成本暴增：多轮任务的交互步数通常是单轮的 3-4 倍，token 消耗 2-6 倍，成本 2-3.5 倍。不是因为任务更难——同样的 verifier——而是因为交互本身的 overhead。

一个有趣的反例：Sonnet 4.6 单轮性能最低（21.3%），但在多轮下只降了 2.5pp——它本来就不擅长自主实现，所以交互式设定对它影响不大。这暗示"交互能力"和"自主实现能力"可能是部分独立的维度。

四、Goal Discovery 的全生命周期分析

论文做了一件很有洞察力的工程：把每个任务的完整需求分解为独立的 sub-goal rubric items，然后在 Agent 的 plan.md 和每个实现 checkpoint 上打分，追踪 Agent 在整个交互过程中对任务目标的理解进展。

关键观察：

  • 最强模型从模糊指令出发仍然能制定出覆盖 >80% latent goals 的计划——这说明"理解用户意图"的前沿模型已经很擅长
  • 但几乎所有模型从 plan 到第一个实现 checkpoint 时 goal coverage 都会下降——把计划变成代码时出现遗漏
  • 经过用户反馈后，GPT 5.5、Opus 4.8、Sonnet 4.6 最终能覆盖 >90% 的任务目标
  • 但 Goal Discovery ≠ Task Success：几乎所有 verifier-pass 的方案 goal discovery score 都在 90%+，但反过来不成立——Agent 可以在"理解所有需求"的情况下仍然写出错误的代码

这揭示了一个重要的断层：理解意图（goal discovery）和正确实现（implementation correctness）是两个不同的失败来源。

五、失败模式分析

论文对 287 条失败轨迹做了语义标注，出了五类失败模式：

失败模式占比说明
Technical Implementation Bug34%需求已知且理解，但最终代码有技术错误
Forgotten Requirement34%早期轮次引入的需求在最终提交中被遗忘/遗漏
Misinterpretation/Bad Assumption14%理解错误或做了错误假设，而非向用户澄清
Missing User Requirement12%用户模拟器从未披露该需求（benchmark 假阴性）
Regression7%新需求导致之前正确的实现被破坏

两个最突出的失败模式——技术 Bug 和遗忘需求——都不是"理解失败"，而是"执行失败"和"记忆失败"。这暗示：即使在需求被充分沟通的情况下，模型在多轮长轨迹中维持实现一致性的能力仍然是一个严重短板。

六、用户模拟器的工程设计：Persona 和模型选择

论文对用户模拟器做了两个关键 ablation：

6.1 Persona 设计的影响

对比 Expert Nitpicker（基于真实数据）vs 中性基线（每次 Agent 请求就给全部需求）：

  • Expert Nitpicker 使交互步数增加 30-50%，用户-Agent 交互轮次增加 90%+
  • 解决率进一步下降——因为用户不会一次性交出所有信息，而是渐进披露、逐步 critique
  • 这验证了 persona 设计对 benchmark 真实性的关键影响

6.2 模拟器模型的影响

对比 GPT 5.5 vs Opus 4.7 作为用户模拟器：

  • GPT 5.5 作为用户更"真实"——需求披露更渐进、workspace 检查更深入（4-5X 的工具调用）、critique 更 detail-grounded
  • 导致 Agent 轨迹更长、交互更多，但最终 Agent 表现更好——因为需求被更完整地传达

这揭示了一个微妙的张力：更好的用户模拟器既让任务更难（更长、更多交互），又让成功更可能（需求传递更完整）。

七、为什么这个 Benchmark 重要

7.1 它测量了一个正交的能力轴

当前所有前沿 benchmark 都在测量同一个维度：给定完整 spec，自主实现。SWE-Interact 测量的是完全不同的东西：在模糊条件下发现意图、在迭代反馈中适应、在需求演化中保持一致性。

这两者可能不是同一个能力的延伸，而是不同的能力维度。证据：单轮最强的模型在多轮下降幅最大。

7.2 它暴露了"记忆"作为一等瓶颈

34% 的失败是遗忘需求。这不是推理失败，不是编码失败，是跨轮次信息保持的失败。当交互轮次从 1 变成 7+，当前轮次的上下文窗口可能已经挤出了早期轮次引入的需求。

这对 Agent 基础设施有直接启示：在长轨迹交互式开发中，需求跟踪和主动回顾（而非仅仅依赖上下文窗口）可能成为关键能力。

7.3 用户模拟器的工程设计本身就是贡献

SWE-Interact 的用户模拟器不是一个简单的 prompt。它是：

  • 基于真实大规模用户数据建模的 persona
  • 有工具访问 Agent workspace 的能力
  • 有渐进式需求披露的策略
  • 最终 verifier 仍然是原始任务的测试套件，而非模拟器的主观判断

这种设计避免了"模拟器说了算"的陷阱——任务成功与否由客观测试决定，模拟器只负责交付需求。

八、局限

问题说明
用户模拟器能力受限当前版本只能运行 git/grep/sed/find，不能修改代码或运行测试。真实用户的交互远比这丰富
单一 persona只建模了 Expert Nitpicker + vibecoding，但真实世界有更多样化的用户类型和交互模式
75 个任务的规模相对较小，且手动筛选了"适合多轮"的任务，可能存在选择偏差
模拟器模型影响 Agent 表现GPT 5.5 和 Opus 4.7 作为用户模拟器产生不同结果——这意味着 benchmark 分数部分取决于模拟器的选择
