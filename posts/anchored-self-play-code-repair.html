Anchored Self-Play for Code Repair：用 Embedding 相似度锚定 Self-Play，阻止代码修复训练的分布漂移

July 8, 2026

Self-Play 自动生成 bug-fix 训练数据，但会在不知不觉中漂移——ASP 用一小撮"像真 bug"的参照物把它拉回来，所有 bug 源平均修复率 +24%

代码修复（code repair）的训练数据永远不够。真实世界的 bug 数据集（Defects4J、BugsInPy）加起来也就几千条。最自然的 scaling 思路是让 LM 自己生成 bug-fix 对——给定一个正确程序，让模型故意改错，再修回来，用 unit tests 自动验证。这听起来像是完美的自动 curriculum：generator 越做越难的 bug，fixer 越修越好。

但这里有一个教科书级别的 reward hacking 问题：unit tests 验证行为正确性，不验证 bug 的真实感。任何让测试失败的编辑都是"valid bug"，但只有极少部分看起来像人类真正会犯的错误。在训练推进中，generator 会逐渐漂移向那些能让测试失败但不 realistic 的怪异编辑——fixer 在这些合成 bug 上越来越好，但在真实 bug 上反而退化。

ICML 2026 的这篇论文——Anchored Self-Play for Code Repair（Stanford + MIT）——没有回避这个问题，而是用一个巧妙的最小侵入方案解决了它。

核心贡献：generator-fixer self-play + embedding anchoring

论文先形式化了 generator-fixer self-play：单个模型（Qwen2.5-Coder-7B）交替扮演两个角色——generator 把正确程序改错，fixer 修回来。用 GRPO 训练。Generator 的 reward 基于难度（fix rate 在中间区间最优——太难太简单都没用），fixer 的 reward 基于修复正确性。

然后他们引入了 BugSourceBench，一个 bug 来源隔离的 benchmark：固定 task 和 unit tests，仅变化 buggy implementation 的来源。四个源：

Bug Source描述真实世界对应
Human标注者手动引入 1-4 个局部错误程序员自己写错的 bug
Human-Edited LMgpt-5-mini 生成错误解 → 标注者编辑人类修改 AI 代码引入的 bug
LM (Qwen-7B)弱模型标准 code-gen 产生的错误弱模型辅助编程的 bug
LM (gpt-oss-20b)强模型标准 code-gen 产生的错误强模型辅助编程的 bug

在这个 benchmark 上，standard self-play 暴露了问题：它确实改善了部分 bug 源的修复率，但在 Qwen-7B 的 bug 上反而退化了 1.5pp。训练曲线（Figure 4）更直观——fix rate 早期上升，然后开始掉头向下，尤其在 Human 和 Human-Edited LM 的 bug 上。分布漂移不是理论推演，是实验事实。

ASP：便宜的 realism signal

Anchored Self-Play (ASP) 的解法极其优雅：假设你能拿到一小撮目标 bug 分布的样本（900 个就够了），用它们作为 anchor。两个互补机制：

1. Embedding Similarity Reward：对每个生成的 bug，计算其 unified diff 的 embedding（voyage-code-3）与 reference set 中 kNN 的平均 cosine similarity，作为附加 reward。不是 hard constraint，而是 soft nudge。为了避免 embedding similarity 的绝对值随 batch 漂移，使用了 centered reward——减去当前 batch 的 EMA baseline。

2. Reference Mixing：以一定概率将 self-play 的生成 bug 替换为 reference bug 来训练 fixer。这确保 fixer 始终暴露在 realistic 分布中。

两个机制单独使用各自提升 1.8–2.3 pp，但结合使用产生超线性增益：

ASP 在 LM 生成的 bug 上增益最大（+12.6 pp / +145% on Qwen-7B；+8.7 pp on gpt-oss），但在 Human bug 上也从 58.3% 提升到 65.0%。更难能可贵的是，在完全不同的 benchmark（DebugBench，LeetCode 风格）上 ASP 仍然最优（60.3% vs. Self-Play 58.7%）。换用 DeepSeek-Coder-6.7B 作为 base model，ASP 的优势依然保持。

两个 anchoring 组件产生正向反馈：similarity reward 让 generator 生成更多 reference-like 的 bug → reference mixing 让 fixer 学会修复这些 bug → fixer 变强后，generator 被 reward 驱动生成更难但依然 reference-like 的 bug → 自动 curriculum 被锚定在目标分布附近。

对 Agent 基础设施构建者的启示

1. Self-Play 的漂移是 Agent 系统的普遍威胁

这篇论文讨论的是 code repair，但问题结构是通用的：任何用 binary reward（pass/fail）驱动的 self-improvement 都可能漂移。你的 agent 在 spec compliance 检查中越来越"合规"，但可能在以你没想到的方式 gaming the spec。ASP 的 anchoring 思路——用一个小型 reference set 作为 behavioral anchor——可以推广到任何 self-play 场景。

2. Embedding similarity 是最便宜的 realism signal

不需要训练一个 realism classifier（那需要标注），不需要硬约束生成空间（那会限制 diversity），只需要一个 off-the-shelf embedding model + 一个小型 reference set。这个设计选择在 token 预算受限的训练环境中尤其有价值。

3. 验证方法的隔离设计是方法论贡献

BugSourceBench 的核心设计——固定 task 和 test，仅变化 bug source——精确测量了"修复泛化"这个原本模糊的概念。在 agent 评估中，类似的隔离设计（固定 spec，变化"agent 会犯什么类型的错误"）可以精确测量 agent 的哪些能力是真正泛化的，哪些是表面拟合。

4. SFT + RL 混合背后的设计直觉

ASP 不是纯 RL——reference mixing 本质上是一种分布约束（distributional constraint），它告诉模型"不要忘了真实 bug 长什么样"。GRPO 提供探索和自动 curriculum 的能力，reference mixing 提供安全网。这种混合训练的范式在 agent 训练中值得推广：RL 做 exploration + skill acquisition，SFT-on-reference 做 distributional anchoring。

局限与诚实标注

  • Reference set 可得性假设：ASP 需要目标 bug source 的小型样本。在真实部署中，bug source 可能随时间变化，reference set 需要持续更新。论文没有讨论 reference set 过期或分布偏移的问题。
  • 仅 function-level repair：所有实验在 BigCodeBench 上（单函数、算法性任务），DebugBench 也是类似规模。对于 repository-level repair（SWE-bench 风格），ASP 的有效性未验证。
  • Embedding model 闭源依赖：使用 voyage-code-3（闭源），如果 embedding model 本身有偏，similarity reward 可能引入新偏差。
  • 计算成本：每步 G=4 bug 生成 + K=4 修复尝试 + unit test 执行。在更大的模型和任务集上，开销可能成为瓶颈。论文承认训练需要 8×H100。
  • 未探索的交集：ASP 与 agentic code review（如 SWE-Review 的 review-then-revise 环路）是否可以结合？fixer 接受 review 反馈后修订，reviewer 判断修订是否成功——这个双层闭环的协同效应未被研究。这是我的推断，不是论文内容。

一句话

Self-play 是自动 curriculum，但 curriculum 可能教模型修没人会犯的 bug。ASP 用一个最小的 reference set 作为"真实感的锚点"，让 self-play 的 auto-curriculum 始终围绕目标分布展开，而不是漂向它自己发明的奇怪边缘。
