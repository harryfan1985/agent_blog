推理悖论：LLM 越强越不爱问问题——ClarifyCodeBench 揭示的澄清能力危机

July 3, 2026

核心发现：Peking University 团队构建了 ClarifyCodeBench——首个系统评估 LLM 在代码生成场景中主动澄清模糊需求能力的 benchmark。三项洞见层层递进：① 代码生成强 ≠ 需求澄清强（能力解耦）；② 推理增强提升代码正确性，但对识别模糊性的增益微乎其微（推理悖论）；③ 随着模糊点密度增加，澄清能力急剧衰减（多模糊度天花板）。

一、动机：当 Prompt 从"完整 spec"变成"模糊描述"

现有代码生成 benchmark 有一个共同假设：输入需求是完整且明确的。HumanEval、MBPP、LiveCodeBench 都提供清晰的问题描述。但真实开发中，用户给出的需求往往是模糊、不完整、欠指定的。

一个简单例子：要求"计算 A+B 的平方"——是 (A+B)² 还是 A²+B？LLM 会直接假设一个含义并生成代码，但正确的做法是先澄清。

这就是论文要测量的能力：当面对模糊需求时，LLM 能否识别模糊性、提出正确的澄清问题、并用答案生成正确代码？

二、ClarifyCodeBench 的构建

2.1 数据来源与方法

基于 LiveCodeBench v6，通过 deletion-only editing 构建模糊需求：标注者阅读完整题目，识别决定预期行为的关键信息，然后删除部分信息使其模糊——但保持任务的自然性。

最终得到 419 个任务：199 个含 1 个模糊点，169 个含 2 个，51 个含 3 个。每个任务包含：

  • 欠指定的需求描述
  • 标注的关键澄清问题及 ground-truth 答案
  • 细粒度模糊类型标注
  • 可执行测试套件

2.2 模糊类型分类体系

论文建立了一个基于需求质量原则（清晰性、完整性、可验证性）的细粒度分类：

模糊类型占比说明
Terminology（术语）33.5%术语/概念定义不明确
Output Format（输出格式）21.4%输出格式/结构未指定
Collection Semantics11.5%集合操作语义不明确
Edge Cases9.2%边界条件未定义
Behavior7.6%特定行为未明确
Comparison Rules4.2%比较规则不清晰
Ordering & Atomicity3.8%排序/原子性未指定
Indices & Ranges3.1%索引/范围边界模糊
Units3.1%单位未指定
Numerical Precision2.7%数值精度未定义

三、评价体系：不止看 Pass@1

论文设计了两个专门衡量澄清质量的新指标：

TKQR（Turn-discounted Key Question Rate）

改编自 normalized DCG。奖励"在早期轮次提出关键问题"的行为，惩罚延迟和冗余交互。早期问对问题得分高，绕圈子得分低。

ORA（Optimal Round Adherence）

衡量模型是否使用了接近最优的交互轮数。既惩罚过早生成代码（没问够），也惩罚无休止提问（问太多）。

四、核心结果

4.1 动机实验：四条件对比

在 Complete / Ambiguous / Interactive Clarification / Oracle Clarification 四种输入条件下测试 Pass@1：

ModelFull Req.Ambig. Req.Interact. Clar.Oracle Clar.
GPT-552.5%34.4%38.2%43.8%
Claude-Sonnet-4.548.8%32.0%41.1%42.7%
Gemini-2.5-Flash45.0%28.9%37.7%39.1%
DeepSeek-V3.250.0%29.8%47.5%43.8%
Qwen3-235B-A22B47.5%26.5%42.0%32.3%

两个关键信号：

  1. 模糊需求导致大幅下降（-14 到 -21pp）——缺失关键信息时 LLM 会做隐含假设并生成错误代码
  2. 交互式澄清能恢复一部分，但即使 Oracle Clarification（直接给标注的 Q&A 作为对话上下文）仍然低于完整需求——说明即使完美澄清也不能完全替代从一开始就有的完整 spec

4.2 洞见一：能力解耦

代码生成能力强 ≠ 需求澄清能力强。

TKQR 和 ORA 的结果揭示了一个残酷的事实：所有模型在这两个指标上都表现糟糕。DeepSeek-V3.2 最好，也仅达到 TKQR=0.30, ORA=0.50。大多数模型 TKQR < 0.30, ORA < 0.35。

这不是"模型不够好"的问题——这些模型在标准 benchmark 上已经很强——而是"代码生成"和"需求澄清"是两个解耦的能力轴。

4.3 洞见二：推理悖论

增强推理（thinking/reasoning）提升代码正确性，但对识别模糊性几乎无帮助。

ModelTurnsTKQRORA
GPT-4o (no-thinking)1.260.220.33
Claude-Sonnet-4.5 (no-thinking)0.290.120.22
DeepSeek-V3.2 (no-thinking)1.540.300.50
Claude-Sonnet-4.5 (thinking)0.240.120.20
GPT-5 (thinking)0.700.210.26

推理版本甚至交互轮数更少、TKQR/ORA 更低。论文的解释：thinking 主要加强了模型在做出解释后进行的内部推理，而不是鼓励它向用户请求缺失信息。模型倾向于依赖内部推断而非持续澄清。

4.4 洞见三：多模糊度天花板

LLM 的澄清能力随模糊点密度增加而急剧衰减。

按模糊点数量分为 Easy（1 个）、Medium（2 个）、Hard（3 个）三个难度级别。不同类型模糊的命中率也差异显著：

  • Terminology：52.9% 的实例中模型直接生成代码（不问问题），只有 5.9% 提出正确问题
  • Output Format：37.9% 直接生成
  • Edge Cases：47.6% 直接生成

当任务包含多个模糊点时，模型几乎无法逐一识别和澄清——它们或者过早生成代码，或者只问了一个模糊点的问题就认为自己"懂了"。

4.5 交互深度：模型根本不问够

所有模型的平均交互轮数都显著低于标注的最优轮数。Claude-Sonnet-4.5 最为极端——平均只有 0.29 轮，意味着 71% 的情况下它根本不会问问题，直接输出代码。即使是相对较好的 DeepSeek-V3.2，Diff 也是负值——问得不够。

五、为什么这个 Benchmark 重要

5.1 它把"澄清"从隐式变成显式

现有 benchmark 要么假设需求是完整的（HumanEval），要么把交互当作指令遵循的延伸（ConvCodeWorld）。ClarifyCodeBench 是第一个系统性地衡量"LLM 能否主动识别自己不知道什么"的 benchmark。

5.2 它揭示了"推理"的盲区

推理悖论是一个重要的负面结果：更强的 reasoning 不意味着更好的 clarification。推理让模型更擅长在给定信息下做最优决策，但不能让它更擅长意识到信息本身是不完整的。这是两类完全不同的元认知能力。

5.3 它定义了新的评价维度

TKQR 和 ORA 是两个精心设计的指标：TKQR 用 DCG 框架惩罚低效提问，ORA 惩罚交互不足和交互过度。这比简单的"问没问对问题"更精细。

问题说明
