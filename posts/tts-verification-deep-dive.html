TTS 深层技术洞察：Verification 的二元张力、三层架构与不可消除的权衡

June 21, 2026

Verification 是 TTS 的神经系统

在测试时扩展（Test-Time Scaling, TTS）的六子类中，Verification 是覆盖率最高的环节——25 篇系统综述论文中 14 篇涉及。这不是偶然。如果把 TTS 比作一个有机体，Parallel Scaling 是肌肉（生成更多候选），Sequential Scaling 是骨骼（结构化推理链），Search 是探索的触角——那么 Verification 就是神经系统。没有它，并行采样是盲目堆砌，串行搜索是无头苍蝇，聚合是随机平均。

本文基于 arXiv:2503.24235（Zhang et al., "What, How, Where, and How Well? A Survey on Test-Time Scaling in Large Language Models"）的系统综述和我们对其的逐层拆解，深入 Verification 的技术内核。

Verification 的四重角色

Survey 原文区分了 verification 在 TTS 体系中的四个截然不同的功能位点：

  1. 选择器（Selector）：在 Parallel Scaling 下，从 N 个候选中选出最优——Best-of-N、Weighted Best-of-N。但"最高分的就是最好的"这个假设本身需要验证。Best-of-N 的性能上限严格受限于 verifier 的区分度。
  2. 引导器（Guide）：在 Sequential Scaling 下，判断当前中间步骤的质量，决定继续修正还是停止，以及往哪个方向修正。Meta-reasoner 用 Bandit 算法动态决定 repeat 次数；Evolving Deeper Thinking 用函数式正确性判断 refinement 的终止条件。
  3. 搜索准则（Criterion）：在 Hybrid Scaling（MCTS/Beam Search）中，每个节点展开都需要 verifier 评分来决定宽展还是深挖。LevinTS（MindStar）甚至把搜索本身建模为计算最优问题——给定置信度要求，最小化期望搜索步数。
  4. 聚合权重（Weights）：Archon 的 Fusion 和 From Drafts to Answers 的聚合微调，都需要 verifier 提供"每个候选解在哪些子问题上可靠"的信息，而不仅仅是全局置信度。

这四重角色中，前两个——Selecter vs Guide——构成了 Verification 最根本的二元张力。

根本不对称：Parallel 中的 Selector vs Sequential 中的 Guide

维度Selector（Parallel）Guide（Sequential）
输入N 个完整候选1 个部分状态 + 历史
输出1 个选择/排名信号：继续/停止/修正方向
时机所有生成完成后（后验）生成过程中（在线）
错误代价选错一个（还有 N-1 个兜底）方向性错误（连锁累积）
核心需求区分度（Discriminability）校准度（Calibration）+ 可定位性

这个不对称有一个残酷的推论：Parallel 对 verifier 的要求比 Sequential 低得多。选择器只需要"A 比 B 好"的相对排序准确即可；但引导器需要绝对判断——"这个状态已经足够好了，可以停了"——这个判断一旦错误，要么过早终止，要么陷入过度修正的 token 消耗黑洞。

然而，所有 TTS 论文的 verification 评估都是作为 Selector 进行的。PRM 在 Best-of-N 中选得准 ≠ PRM 能精确指出"第 3 步需要修正"。这是当前研究最大的隐性前提错误。

Verifier 类型谱系：Outcome vs Process

Survey 将 verifier 分为两大类型，共 20 种具体技术：

Outcome Verification（12 种）——只看结果

  • Naive ORM：训练分类器对完整答案打分。最简单，但无法区分"推理过程全错蒙对答案"和"真会了"。
  • Heuristic Verifier：领域规则检查（数学答案格式、代码语法）。唯一在两个范式中都有确定性指导能力的 verifier——不是概率，而是"这里错了，因为违反规则 X"。
  • Functional Verifier：函数式正确性判断——travel plan 是否超预算？约束是否满足？在 Sequential 中有独特优势：天然提供可定位反馈（"约束 3 被违反"），直接告诉模型该改哪里。
  • Bandit Verifier（Meta-reasoner）：唯一把验证对象从"输出"提升到"元策略"的 verifier。不是判断答案对不对，而是判断"对这个难度的问题，当前 refine 策略够不够？"
  • Generative Verifier：利用 LLM 的生成能力做验证——把验证重构为 next-token prediction 任务。
  • Multi-Agent Verifiers：多个 agent 从不同角色/维度独立验证。在 Sequential 中有独特价值：agent 间的分歧本身就是最强的修正信号。
  • Unit Test Verifier：代码执行测试。和 Heuristic 一样提供确定性+可定位反馈。
  • XoT / WoT：多视角验证（Assertion, Process, Result 三个角度）。
  • Self-Reflection Feedback：将反馈利用形式化为优化问题，在 test-time 求解。
  • Discriminator：SFT 一个领域特定的判别器 LM。
  • OVM：在 outcome supervision 下训练 value model，用于 guided decoding。

Process Verification（8 种）——逐步评分

  • Naive PRM（Lightman et al. 2023, Snell et al. 2024）：对推理链的每个步骤独立评分。训练为二元分类器，Monte Carlo rollout 软标签。
  • State Verifier（ToT）：评估推理树中间状态，指导 BFS/DFS 搜索方向。Sequential 中最纯粹的 Guide 角色。
  • Deductive PRM：不是每一步都验证，而是识别推理链中的关键断言，只验证那些——大幅降低验证成本。
  • Self-Evaluation：同一模型评估自己的输出。最大问题是系统性自我偏好——倾向于给自己刚生成的答案打高分。
  • Tool-Grounded Verifier：代码执行器、形式化证明器。Sequential 中最强的 Guide 信号——给出的不是"0.6 分"而是 NameError: 'x' is not defined，直接指向精确修正。
  • V-STaR：在正确和错误的自生成数据上联合训练，增强区分度。
  • Lego-Prover：定理证明中的 lemma 级验证。
  • PoT：将计算步骤委托给外部语言解释器验证。

PRM 的意外发现：Last-Step 聚合最优

Snell et al. (2024) 比较了三种 PRM 聚合策略：

策略方法效果
Min取所有步骤最低分次优
Prod所有步骤概率乘积次优
Last仅用最后一步分数最优

这个结果极其反直觉：PRM 被设计为逐步验证，但最后一步分数反而最准。这意味着 PRM 优于 ORM 不是因为它"逐步验证"，而是因为逐步训练的表示学习效应让模型学到了更丰富的内部表征。PRM 训练过程本身就是优势，而非逐步评分机制。

这对整个 PRM 研究方向提出了一个尖锐的问题：如果 last-step 聚合最优，那训练 PRM 的成本（步骤级标签、MC rollout）相对于直接训练 ORM，到底值不值？FineVerify（arXiv:2606.00660）的细粒度验证可能提供了部分答案——不是逐步评分，而是把问题拆成子约束逐条验证。

Agent 验证的三层架构

将 Verification 放到真实 Agent 管道中，我们发现一个三层防御体系：

Layer 1：即时验证（Per-Action）

每个工具调用后立即执行。验证器全部是确定性规则（非 ML）：工具错误码、格式检查、单元测试。成本 ≈ 0，延迟 < 100ms。失败 → 重试（修正成本最低）。盲区：无法检测语义错误（格式正确但内容错误）。

Layer 2：步骤验证（Per-Subtask）

每个子任务完成后执行。需要语义理解——不是"格式对不对"而是"意思对不对"。使用 ML-based verifier：PRM、State Verifier、Self-Evaluate、Multi-Agent Verifier、FineVerify 细粒度验证。成本 = 1~N 次 LLM 调用。盲区：无法检测跨子任务的隐藏因果链错误。

Layer 3：轨迹验证（Per-Trajectory）

整个 Agent 执行完成后，从用户意图层面判断。使用 ORM、Multi-Perspective Verifier、Generative Verifier。面临最根本的挑战——没有 ground truth。用户是否满意是主观的。UNSATISFIED → 全部重做（修正成本极高）。

不可消除的权衡：信号强度 vs 语义深度

信号强度（客观性）
    │
 L1 ●●●●● Tool error / Heuristic / Unit test  ← 最强信号，最浅语义
    │                                          "JSON 格式正确"不代表内容正确
 L2 ●●●○○ PRM / Self-Eval / MAV              ← 中等信号，中等语义
    │
 L3 ●●○○○ ORM / LLM-as-Judge                  ← 最弱信号，最深语义
    │                                          "用户满意吗"没有客观答案
    └──────────────────────────────────────────→ 语义深度

这不是工程问题——这是认识论限制。越接近最终目标（"用户满意"），验证就越主观；越接近底层执行（"函数调用成功"），验证就越客观但与最终目标越脱节。三层架构不是要消除这个权衡，而是让每一层做它最擅长的事。

关键推论：系统设计应该最大化 L1 和 L2 的覆盖，最小化对 L3 的依赖。因为 L3 修正成本极高（全轨迹重做），而 L1 修正成本极低（重试一个动作）。

L2.5：错误归因——最大的研究空白

在 L2（子任务失败）和全局 Replanning 之间，缺少一个关键步骤：错误归因。子任务失败是因为：

  • 规划本身有问题（子目标不可行）→ 需要全局重规划
  • 执行有问题（选错了工具/参数）→ 局部修正即可
  • 环境变化（前提条件不再成立）→ 需要重新感知

当前没有任何 TTS 论文处理这个归因问题。Agent 系统在子任务失败时要么盲目重试（L1 思维），要么完全重规划（L3 思维）。这导致大量不必要的计算浪费。

当前研究坐标：0/25

TTS survey 覆盖 25 篇论文，0 篇在真实 Agent 管道环境中评估过任何层级的 verification。所有评估都是在数学/代码基准的最终答案正确率上（等价于只评估了 Agent 的 L3）。对 L1/L2 的独立价值、verifier 的校准度（Guide 角色的核心需求）、跨子任务因果链的错误传播——完全空白。

同时，Agent 场景天然拥有的环境反馈（工具执行结果）是最强的免费验证信号——比 PRM 的软标签精确得多。但在 TTS 文献中，Tool-Grounded Verification 仅出现在 2/25 篇论文中（START, MA-LoT），且都评估在数学/代码而非 Agent 任务上。这是当前 TTS 研究中理论与实践之间最大的效率差。

设计启示

  1. L1 应该全是确定性规则——不依赖 ML 模型，成本零、延迟零、零误报。
  2. L2 是 TTS 研究可以立即迁移的——PRM、State Verifier 在数学推理中的逐步验证经验，可改造为 Agent 子任务验证。
  3. L2.5 错误归因是最急迫的未解决问题——需要系统化机制区分规划错误 vs 执行错误。
  4. L3 需要从"评分"转向"约束满足"——FineVerify 的细粒度约束验证是正确方向。约束不满足 → 知道缺了什么 → 可以定向修正而非全盘重做。
  5. 环境反馈是 Agent 的免费 PRM——每次工具调用都在产生高质量验证信号。Agent 框架应该优先利用环境反馈，仅在环境反馈不可用时才 fallback 到 ML verifier。

技术声明：本文技术分析基于 Zhang et al. (2025) "What, How, Where, and How Well? A Survey on Test-Time Scaling in Large Language Models" (arXiv:2503.24235) 全文，以及 Snell et al. (2024) "Scaling LLM Test-Time Compute Optimally" (arXiv:2408.03314)。三层 Agent 验证架构、L2.5 错误归因概念、及信号强度-语义深度权衡分析为作者基于综述文献的综合推演。
