Osmani 三层体系：Factory → Harness → Loop

June 10, 2026
"Cherny's point isn't that the work got easier. It's that the leverage point moved. Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go."
— Addy Osmani, Loop Engineering

三层体系全景

Addy Osmani（Google Chrome engineering lead）在 2026 年 2-6 月连续发表三篇文章，构成从宏观叙事到微观实现的完整认知栈。三层不是替代关系，是从 vision → discipline → operation 的递进。

Factory Model（2 月）说：你不再写代码，你建工厂。Harness Engineering（4 月）说：工厂里的每个工位你都得亲手配置，每条规则必须 earned from failure。Loop Engineering（6 月）说：配置好了之后，让系统自己运转——但三类风险随之变尖锐。

三层核心对比

  • Factory Model — 战略层：范式描述，"世界变了"。Bottleneck = 认知层面。隐喻 = 工厂。
  • Harness Engineering — 战术层：工程纪律，"怎么应对变化"。Bottleneck = 配置层面。隐喻 = 棘轮。
  • Loop Engineering — 调度层：自主运转，"怎么跑起来"。Bottleneck = 流程层面。隐喻 = 心跳。

Osmani 自己的话："Loop engineering sits one floor above the harness. The harness but it runs on a timer, it spawns little helpers, and it feeds itself."

Factory Model：建工厂的人

三代 AI 编码工具的演进：

  • Gen 1：Autocomplete — 减少击键，workflow 不变。你驱动，工具辅助。
  • Gen 2：Synchronous agent — 自然语言描述，你仍每步在场。你协作，agent 生成。
  • Gen 3：Autonomous agent — 给 spec 跑数小时，回来看结果。你编排，agent 自主执行。

工厂隐喻的五个映射：Quality control → verifier sub-agent，Process documentation → Skills/AGENTS.md，Precise inputs → Spec，Reliable environment → Sandbox，Stalls when unreliable → Flaky test 在 40 agent 并行时变成系统性阻塞器。

核心洞见：Spec 是 leverage 放大器。烂 spec × 50 agent = 50 个方向各异的烂实现。好 spec × 50 agent = 50 个一致的精准实现。"Vague thinking does not just slow you down. It multiplies."

但 Factory Model 同时指出：验证是未解决的问题。"The unsolved problem is verification, not generation."——语气是无奈的，五个具体难点被列出但无解决方案。

Grady Booch：软件的第三个时代

Osmani 引用了 Grady Booch——UML 和 OOAD 先驱——的判断："We are living through software's third age — a new golden era defined by rising abstraction, where the job of the developer shifts from writing instructions to defining intent."

这个判断提供了历史纵深：从汇编到 C，从 C 到托管语言，从托管语言到框架和云基础设施——每一步都曾让人觉得 "coding 变简单了"。现在这一步也一样。

为什么测试现在比以往更重要

Factory Model 中最强的实践主张：Red/Green TDD 在 agentic workflow 中从 "good practice" 升级到 "接近 mandatory"。

原因是 scale 效应：一个开发人员跳过 test-first 的代价可控；50 个 agent 同时跳过 test-first 的代价是指数级的——agent 会找到各种方法 "pass tests"，如果 tests 是 implementation 之后写的，它们验证的是 "代码做了什么" 而不是 "代码应该做什么"。"Red/green TDD is a shorthand every good model understands."

验证的五个具体难点

Osmani 列出的五个问题是精确的工程诊断，不是 handout：

  1. Pass 的测试不一定 catch regression
  2. Agent 可以写 technically valid 但 miss 重要 case 的测试（自查无门）
  3. UI 验证仍然脆弱——视觉和行为回归难以自动检测
  4. Context window 限制导致 agent 遗漏不在当前窗口内的约束
  5. Flaky environment 在 40 个 agent 并行时从 annoying edge case 变成系统性阻塞器——工厂停转

Harness Engineering：棘轮模型

核心公式：Agent = Model + Harness。如果你不是模型，你就是 harness。

关键数据：Viv Trivedy 团队仅改 harness，Terminal Bench 从 Top 30 → Top 5。同一模型（Opus 4.6），不同的配置，截然不同的表现。这直接反驳了 "等下一个更好的模型" 的叙事。

棘轮模型：每次 agent 出错 → 分析根因 → 写进 harness → 永不再犯。不是等下一个版本修复，是你自己动手堵漏洞。

每条 AGENTS.md 规则必须earned from failure history——不是 brainstorm 出来的，是从 failure history 挤出来的。HumanLayer 的规则：< 60 行，pilot's checklist not style guide。

与 Factory 不同的是验证态度：验证是可设计的目标。"Hooks are what separate 'I told the agent to do X' from 'the system enforces X.'"——语气是工程性的，不是无奈的。

Harnesses don't shrink, they move

Anthropic 的 harness 文章中最耐人寻味的观察：模型变好，harness 不消失——它移动。

Naive 的叙事是 "更好的模型让 harness 过时"。但实际情况是：Opus 4.6 基本消灭了 context-anxiety（Sonnet 4.5 在接近 context 上限时会过早结束任务），所以焦虑缓解 scaffolding 变成了死代码。但天花板随模型一起升高——原来不可及的任务现在在射程内，产生了新的失败模式。焦虑 scaffolding 消失了，替代它的是 multi-day memory policy、或协调三个专精 agent 的 harness、或 UI 设计质量的 evaluator。

"Every component in a harness encodes an assumption about what the model can't do on its own."

Model-Harness 训练耦合

Viv Trivedy 命名了一个重要现象：现代 agent 产品是在 harness 的闭环中进行 post-training 的。模型专门优化 harness 设计者认为它应该擅长的操作——文件系统操作、bash、规划、subagent dispatch。这就是为什么 Opus 4.6 在 Claude Code 内部感觉不同：co-training 创造了过拟合。

实践含义有两层：1) harness 是活的系统，不是一次性配置；2) "最好"的 harness 不一定是模型被训练的那个——可能是为你的 task 设计的那个。Top 30→Top 5 的跳跃是最清晰的 proof point。

Harness-as-a-Service (HaaS)

Viv Trivedy 的另一个贡献：从 LLM API（给你 completion）到 Harness API（给你 runtime）的转变。Claude Agent SDK、Codex SDK、OpenAI Agents SDK 都指向同一个方向。

"The default path used to be: build your own loop, wire up your own tool-calling, handle your own conversation state, invent your own approval flow. Now the default path is: pick a harness framework, configure it along the four pillars (system prompt, tools, context, sub-agents), and put the rest of your effort into domain-specific prompt and tool design."

Loop Engineering：自主运转的系统

Osmani 的转向："Loop engineering is replacing yourself as the person who prompts the agent. You design the system that does it instead."

Loop 的六个组件：

  • Automations：cron 或 /goal（run until condition met）。/goal 的关键设计——独立的小模型检查完成度。
  • Worktrees：并行 agent 的物理隔离。两个 agent 写同一个文件 = 两个工程师 commit 同一行。
  • Skills：项目知识外部化——intent written down on the outside，每次 run 都读到。
  • Connectors：MCP 接入外部工具——让 agent 从 "说能做什么" 变成 "真的做"。
  • Sub-agents：制造者/检查者分离——不同 instructions，有时不同 model。
  • State file：跨 run 持久化状态——下次 run 从上次停的地方继续。

关键发现：Codex 和 Claude Code 这五个组件的能力本质相同，差别仅在命名和配置格式。Loop 的设计不绑定任何产品——是架构模式而非产品特性。

但 Loop 带来三类随 loop 变尖锐的风险：

  • Verification still on you：无人值守 = 无人值守犯错。"its done" 是 claim 不是 proof。
  • Comprehension debt：loop 越快 ship 你没写的代码，理解差距越大。
  • Cognitive surrender："same action, opposite result"——设计 loop 用判断力是解药，用来逃避思考是毒药。

/loop vs /goal：两种自主 primitive

Osmani 特别区分了两种自主触发模式，两者的工程语义完全不同：

  • /loop：按时钟走——"每隔 N 分钟重跑"。适合周期性发现任务（triage、CI 监控、commit briefing）。每次 run 独立。
  • /goal：按条件走——"一直跑到条件满足为止"。适合目标驱动的实现任务。关键设计：每个回合结束后，一个独立的小模型检查是否完成，不是写代码的模型自评。"all tests in test/auth pass and lint is clean"——设定好就走开。

Codex 和 Claude Code 对这两个 primitive 的实现完全相同——名称相同、语义相同。这不是巧合，是工程上的收敛。

Codex vs Claude Code：六个组件的实现对照

Osmani 在 Loop Engineering 中给出了一个精确的对照表，证明了"这不是工具之争，是架构模式之争"：

  • Automations：Codex = Automations tab → Triage inbox。Claude Code = /loop + /goal + hooks + GitHub Actions。
  • Worktrees：Codex = Built-in per thread。Claude Code = git worktree + --worktree flag。
  • Skills：Codex = SKILL.md, $name 调用。Claude Code = SKILL.md。
  • Connectors：Codex = MCP Connectors + Plugins。Claude Code = MCP Servers + Plugins。
  • Sub-agents：Codex = .codex/agents/ TOML。Claude Code = .claude/agents/ + agent teams。
  • State：Codex = Markdown / Linear via connector。Claude Code = AGENTS.md + progress files / Linear via MCP。

"The names are a bit different here and there but the capability is the same thing. And once you notice the shape is the same you stop arguing about which tool, you just design a loop that still works no matter which one you happen to be sitting in."

一个完整的 Loop 长什么样

Osmani 给出了一个 concrete example：

每天早上 automation 在 repo 上运行 → 调用 triage skill（读昨日 CI failures、open issues、recent commits）→ 写入 state file 或 Linear board。对每个值得修复的发现 → 开独立 worktree → sub-agent A 起草修复 → sub-agent B 对照 project skills 和已有 tests 审查 → connector 开 PR + 更新 ticket。无法自动处理的项目进入 triage inbox 等人工。State file 记住被尝试过的、通过了的、仍 open 的。明天早上从今天停的地方继续。

"You designed it one time. You did not prompt any of those steps."

Intent Debt：技能解决的根本问题

Osmani 引入了一个概念——intent debt。Agent 每次启动都是冷启动，你未写明的意图会被模型用自信的猜测填补。Skill 是把意图写在外部，agent 每次 run 都读到。

"A skill is how you stop re-explaining the same project context every session like a goldfish. Skills are where intent stops costing you over and over. Without skills the loop re-derives your whole project from zero every cycle, with skills it kind of compounds."

Orchestration Tax：工具消除机械冲突，你才是上限

Worktree 消除了并行 agent 的文件碰撞——但 你的 review bandwidth 才是真正的上限。不是工具限制你能并行多少个 agent，是你的 review 能力。

"The worktrees take away the mechanical collision but YOU are still the ceiling, your review bandwidth decides how many you can actually run, not the tool."

Spec 的三层定位

同一样东西（AGENTS.md / SKILL.md），三个视角：

  • Factory：Spec = 放大器。质量决定 leverage 上限。缺失后果：50 agent 同时偏航。
  • Harness：Spec = 止损器。每条规则 earned from failure。缺失后果：agent 重复犯同一个错。
  • Loop：Spec = 累积器。每次 automation 触发都读取同一份 skill，跨 run compound。缺失后果：每个 cycle 从零推导项目惯例。

"Without skills the loop re-derives your whole project from zero every cycle, with skills it kind of compounds."

验证的三层态度

  • Factory：验证是未解决的问题。语气 = 无奈。
  • Harness：验证是可设计的目标。语气 = 工程。
  • Loop：验证是必要但非充分条件。语气 = 警觉。

三层逐级加剧了同一组风险。Factory 告诉你 leverage 变大了 but 验证跟不上。Harness 告诉你需要 discipline but 每条规则必须 earned。Loop 告诉你 agent 可以自主跑 but 认知投降和理解债的风险比手动 prompt 时更大。

与 OpenSkill 的交叉

Osmani 三层体系和 OpenSkill（arXiv 2606.06741）在概念上互补：

  • OpenSkill 产Skills（知识 artifact）——部署前构建。验证用 virtual verifier（自建锚点测试）。
  • Loop Engineering 产Loops（自主运行的流程）——部署后利用 skill 的自主调度。验证用 sub-agent（不同模型检查）。

一个端到端想象：OpenSkill 生成的 skill → 注入 skill 目录 → 由 Loop Engineering 的 automation 调度 → sub-agent 执行 → verifier 检查 → connector 开 PR。OpenSkill 解决上游（如何自动造 skill），Loop 解决下游（如何调度、组合、执行 skill）。

结论

Osmani 三篇文章共享同一个核心信念：Leverage point moved, not eliminated。自动化放大的是你的判断力，不是替代你的判断力。判断力好的人用 loop 加速，判断力差的人用 loop 逃避——工具分不出这两者，但结果天差地别。

Factory 版："coding changed dramatically, software engineering at its core has not." Harness 版："a decent model with a great harness beats a great model with a bad harness." Loop 版："Cherny's point isn't that the work got easier. It's that the leverage point moved."

三层缺一不可。没有 leverage 的 discipline 是苦力，没有 discipline 的 leverage 是灾难，没有两者就不需要 loop——因为你仍然坐在那里手动 prompt。
