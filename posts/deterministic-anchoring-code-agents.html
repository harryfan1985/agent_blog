确定性锚定效应：Code Agent 不需要更聪明，只需要更有纪律

July 3, 2026

核心论点：对 grep-first 代码 Agent，静态结构的主要价值不是让 Agent "更聪明"，而是让它的导航变得 disciplined and reproducible（有纪律且可复现）。这一论断来自 Beihang University 发表在 ISSTA 2026 的研究，通过系统性地向代码库注入静态分析标注，量化了"确定性锚定"对 Agent 定位精度、轨迹行为和运行间稳定性的影响。

一、问题：最强的 Agent，最盲的导航

现代代码 Agent 的王者架构是 grep-first：Codex、Claude Code、SWE-agent，都是同一个模式——发 keyword query（grep/rg）→ 看匹配结果 → 推理 → 再搜。这套方案简单、语言无关、对 broken code 鲁棒，benchmark 成绩已经很强（SWE-bench Lite Func@5 ≈ 83.2%）。

但它有一个根本性的 representation mismatch：

Agent 的检索方式：关键词匹配（grep/rg）
代码的组织方式：调用图、继承链、配置传播、数据流

Grep 返回的是按词法相似度排名的代码行，不暴露 谁调用谁、哪个配置值传播到了哪个模块、哪个类继承自哪个基类。Agent 必须通过 ad-hoc 多跳搜索来重新发现这些本来就存在于代码库中的结构关系。

更致命的问题是，LLM 控制器的 随机性 与这种结构盲检索耦合后，会产生脆弱的轨迹：同一个 bug，两次运行可能访问不同的文件、走不同的路径、产出不同的 patch——即使最终结果相同。对工程师来说，无法预测的行为意味着调试困难，失败难以复现。

二、核心洞见：确定性锚定效应

论文的核心发现被称为 Deterministic Anchoring Effect（确定性锚定效应）：

静态结构帮助 Agent 的主要机制不是提升"智能"，而是通过提供确定性路标，约束概率性探索，使导航变得有纪律且可复现。

这个区分很关键：传统思路是"让 Agent 理解更多代码语义"，论文的思路是"让 Agent 的行为更可预测"。两者不互斥，但后者是一个被严重低估的维度。

三、CodeAnchor：把结构 bake 进代码

具体做法是 CodeAnchor 框架：

  1. 离线静态分析：用 PyCG（调用图）+ AST 提取器，分析调用关系、继承关系、数据流、配置使用
  2. 生成结构化注释：将关系编码为紧凑的 tag，注入源代码中对应实体附近
  3. 运行时被动消费：Agent 继续用 grep 搜索，搜索结果中同时包含原始代码和附近的 tag

Tag 的格式很简单：

# === Function Annotation ===
# function _get_locale_dirs
# used by: fetch, lang_stats, update_catalogs
# === Function Annotation ===
def _get_locale_dirs(resources, include_core=True):

关键设计决策：不新增 MCP 工具，不修改 agent loop。论文的 pilot 实验验证了，即使提供显式 call-graph 工具，Agent 也不爱用——所以"被动注入"比"主动工具"更有效。

五、关键结果

RQ1：拓扑结构有用吗？——有用，且稳健

Func@5 Δ+2.2pp (p=0.041)+1.2pp

出现了 structural detour 现象：File@1 微降（-1.5pp），因为 Agent 先访问 tag 指出的 hub/helper 文件，但最终 Func@10 提升——这是"用结构化路径换运气式词法命中"的 trade-off。

下游修复实验更说明问题：在 Baseline 和 Anchor-Topo 的 top-5 函数集不一致的 80 个 Verified 实例上做受控修复实验，Anchor-Topo 修了 48/80 vs Baseline 38/80（+12.5pp），且是 strict superset——Baseline 修了的 Anchor 全修了，外加 10 个。

RQ2：粒度和方向性——依赖仓库规模

两个核心发现：

  • 粒度饱和：Anchor-Dense 的额外数据流/配置边在 Lite 上不提升准确率，反而 +4.9 rounds 和 +18.8% tokens。只在 3/274 (Lite) + 15/500 (Verified) 的隐式依赖案例中救场。
  • 方向性依赖规模：中型仓库（avg 35k LOC）Anchor-Topo 双向链接最优；大型 hub-heavy 仓库（avg 120k LOC, 23% hub 节点）Anchor-Inv 反向链接更优。原因是 forward edges 在 hub 节点上产生大量重复 tag 文本，把 agent 引向 utility 模块而非 bug 现场。

论文给出了可操作的部署启发式：

avg_out_degree < 5  → Anchor-Topo（全拓扑）
avg_out_degree > 8  → Anchor-Inv（仅反向）
中间地带            → Anchor-Topo + degree capping

方差减半是这篇论文最实质性的贡献。不是"平均更好"，而是"更可预测"。Pass@1 +3.4pp 意味着单次运行的可靠性提升——开发者通常只跑一次，这比 Pass@10 的微弱提升重要得多。

55-59% 的任务中至少有一次 tag-guided hop 命中 ground truth。Tag 不是被无视的噪音，而是被真正用来导航的确定性信号。

六、深层分析：为什么重要

1. 它回应的不是"能不能更好"，而是"为什么不够好"

在 grep-first agent 已经 83%+ Func@5 的 regime 下，accuracy headroom 极其有限。论文的论点是：真正的瓶颈不是 accuracy，而是 stochasticity。方差减半 + Pass@1 提升，才是生产级部署的真正需求。

2. 它重新定位了静态分析的工程价值

传统静态分析追求 soundness 和 completeness——这是学术圈的执念。CodeAnchor 证明：unsound、不完整的静态分析仍然对 LLM agent 有巨大价值。因为 LLM 把 tag 当 soft hint 而非 hard constraint——缺少一条边不会导致崩溃，Agent 会 fallback 到词法推理。

这是"不完美的结构信息仍然有用"的工程证明。

3. Stability 应该成为一等指标

当前 code agent benchmark 几乎只看 Pass@k 的平均值。论文把 run-to-run variance 和 Pass@1 作为核心指标，更贴近实际部署场景。当工程师看到一个失败时，他们需要知道"这是系统性缺陷还是随机噪声"——如果两次失败走了同样的路径，缺失的步骤一目了然；如果每次失败路径都不同，每个失败都是全新的调查。

七、与 Agent 基础设施的关联

对于 Agent 基础设施/Harness Engineering 方向，这篇论文有几个直接启示：

  • Agent harness 应提供 codebase 预处理 pipeline：CodeAnchor 的离线 tag 生成是 task-agnostic 的，在 agent 启动前完成。当前主流 harness 在这一点上是空白的——agent 每次从零开始 grep。
  • "被动注入"优于"主动工具"：把结构信息 baked into the environment，而不是 add as tools。Agent 不擅长主动使用 graph 工具，但能被动利用 inline 的结构信息。
  • 轻量静态分析的高 ROI：不需要 sound analysis，不需要 fancy graph DB，用 conservative PyCG + AST 就能拿到显著收益。这对 agent 基础设施团队意味着：低成本、高回报的静态分析预处理，值得内置到 harness 中。

八、局限

问题严重程度
仅 Python — tag schema 是语言无关的，但实验只覆盖了 Python中等。Java/C++ 的静态分析工具链更成熟，但效果需验证
仅一个 Agent (Codex) — 未测试 Claude Code/Cursor 等中等。作者承认这是外部有效性的主要威胁
仅 localization — 端到端 repair 结果仅在 differential subset 上做了受控实验低-中。Differential 实验显示强传播（+12.5pp repair）
SWE-bench 的 fault depth 有限 — 多数 1-2 hop中。作者假设更深的多跳 fault 会受益更多，但缺乏 benchmark

论文明确声明了三个未经验证的范围维度：其他 agent 架构、其他语言、更深的结构性故障——这三个是明确的未来工作方向。
