OKF：Google 把 LLM-wiki pattern 变成开放标准——让知识从"模型内部状态"变成可审计的外部 artifact

July 13, 2026

OKF：Google 把 "LLM-wiki pattern" 变成了一个开放标准——让知识从"模型内部状态"变成可审计的外部 artifact

AI agent 有一个共同的需求：它们需要知道"事实是什么"。不是 RAG 检索到的语义相似片段——那种东西可能对也可能错——而是已经被验证的、可被精确引用的知识。在 agent 内部，这通常是一堆 Markdown 文件加 YAML frontmatter——一个 LLM-wiki。每个认真的 agent 团队都在维护这种东西。Obsidian、Notion、纯 Git 仓库——形式不同，本质相同。

问题是：每个团队都在发明自己的格式。没有统一的 type taxonomy、没有标准化的 frontmatter 字段、没有跨工具的互操作性约定。你的 agent 的知识库和我的 agent 的知识库用的是完全不同的 schema——即使它们描述的是同一个东西。

Google 在 2026 年 6 月发布了一个极简但精准的答案：Open Knowledge Format (OKF)。不是另一个知识管理平台——只有一个 spec。三个核心约定：YAML frontmatter + Markdown body + Git 仓库。

OKF 极简到什么程度

整个 spec 的核心是这些：

---
type: BigQuery Table            # REQUIRED — 概念类型
title: Customer Orders           # 可选 — 显示名称
description: One row per order  # 可选 — 一句话摘要
resource: https://console...    # 可选 — canonical URI 锚定
tags: [sales, orders]           # 可选 — 标签
timestamp: 2026-05-28T14:30:00Z # 可选 — 最后修改时间
---

# Schema
| Column | Type | Description |
|...

# Citations
[1] [Source URL](https://...)

就这些。没有 schema registry、没有中心化 authority、没有必须安装的 SDK。用 cat 就能读，用 git clone 就能分发。如果你已经有了 Obsidian wiki——你已经有了 OKF 的 80%。

但 OKF 多出来的 20% 恰好是 agent 评测最需要的。

OKF 在 agent 评测上六个 RAG 做不到的优势

1. 可验证性：每条知识都有地面真相引用

RAG 检索到一个 chunk，你怎么验证它是否正确？你只能再跑一次 embedding 搜索。OKF 的 resource 字段直接指向 ground truth——"这条知识描述的是 BigQuery 上的那个表，它的 schema 就在那里"。Agent 引用是否正确，不需要另一个 LLM judge 来猜。

在 AgentLens（trajectory evaluation）的五维体系里，OKF 直接让 INSTRUCTIONCOMPLIANCE 可测量：agent 要么正确引用了 resource，要么没有。

2. 召回率可精确计算

一个 OKF bundle 是一个封闭世界——所有概念都在目录里。你可以精确测量：

  • "这个 bundle 有 163 个实体，agent 在 100 次查询中正确引用了多少个？"
  • "哪些 concept 从未被 agent 引用过（orphan knowledge），需要改进检索策略？"

RAG 永远做不到这一点——你永远不知道"没检索到的 chunks 里是否有更好的答案"。

3. 知识漂移可追踪

log.md + timestamp + git history 构成完整时间线。你可以精确测量：

  • 某个 concept 更新后，agent 的行为是否退化
  • 知识漂移 → agent 行为漂移的因果关系

这正是 AgentLens 的 nightly regression pipeline 需要的——不是"agent 有没有变差"，而是"因为什么知识的变更导致 agent 变差"。

4. 跨 agent 知识一致性比较

因为 OKF 是开放格式，不同 agent（Claude Code、Codex、你的 agent）可以共享同一份 knowledge bundle。消除 knowledge source 的 confounding 因素后，agent 之间的比较才是真正的 agent-to-agent comparison，不是"模型 + 知识库"的混合物。

5. Side-by-side 审计

两个 agent 对同一份 OKF bundle 的修改可以直接 git diff——比比较"两个 agent 的聊天记录"精确得多。AgentLens 的 pairwise review 如果有 OKF diff 作为补充证据，就不再完全依赖 LLM judge 的主观判断。

6. Agent→知识 的正确性可自动验证

OKF 的 Citations 章节让每个 claim 都有外部锚点。结合 VLP 的 API verifier 或 Aria 的 Coq harness 的设计模式——你可以写一个自动验证器：

  • Agent 声明"Customer Orders 表有 total_usd 列" → 验证器检查 OKF concept 的 resource URI → 确认
  • Agent 引用了一个不存在的 concept → 验证器返回 broken link

这是从"相信 agent 说的"到"验证 agent 说的"的关键跃迁。

OKF 与你的体系的精确对应

你的 wiki（Obsidian）OKF差异
entities/Foo.md + frontmatterconcept document, type: entity缺少 resource anchor
concepts/Bar.mdconcept document, type: concept缺少标准化 type taxonomy
sources/YYYY-MM-DD_Paper.mdconcept document, type: source缺少 citations 章节
index.md（统计 + 分类列表）index.md（渐进式披露）格式几乎相同
Wikilinks [[Foo]]Markdown links [Foo](/path.md)OKF 用标准 Markdown，更可移植
MCP/Obsidian 可读任何 cat 可读OKF 消除工具锁定

你的 wiki 到 OKF 的迁移成本接近零——主要差异是加 resource 字段（把 entity 锚定到 ground truth URI）、加 citations 章节（让每个 claim 可追溯）、而 Wikilinks 需要转换成标准 Markdown links（或保留双格式兼容）。

局限：OKF 不解决什么

  • 仅知识表示层：OKF 是文件格式，不是检索系统。你需要自己搭索引、搜索、embedding。
  • 封闭世界假设：召回率精确可测量这个优势有一个前提——bundle 是完整的。OKF 不会告诉你"你还缺了什么知识"。
  • 没有语义搜索：没有内置 embedding 检索。你在图中的 Router 流正是 OKF（canonical）+ RAG（exploratory）互补的依据——两者各自解决对方不能解决的。
  • v0.1 极其极简：没有 schema registry、type taxonomy、查询语言。这是刻意为之（保证可移植性），但意味着评测基础设施需要自己构建。
  • Google 推动但生态初生：2026 年 6 月发布，主要 adoption 在 Google Cloud 生态内。

OKF + RAG 的互补性

回到你图中的 Router 设计——

查询类型OKF（Wiki）RAG（Vector DB）
确定性事实✅ resource 锚定，可验证❌ 语义相似 ≠ 正确
封闭世界召回率✅ 精确可计算❌ 无法计算"遗漏了什么"
跨 agent 可比较✅ 同一 bundle❌ chunking 策略影响结果
开放性探索❌ 只有已知概念✅ 可以发现未结构化的关联
快速原型❌ 需要人工整理✅ 零准备即可检索

OKF 和 RAG 不是竞争关系——它们在你的 Router 设计中正好是互补的两路：OKF 回答"已知的确定性事实"，RAG 探索"未知的开放性问题"。评测上的优势来自前者，覆盖度上的优势来自后者。

一句话

OKF 的价值不在于它是一个更好的知识管理工具——它是对 agent 评测最稀缺的东西（可验证性、可比较性、可追溯性）的一个极简规范。它把知识从"模型内部状态"变成了"可被自动验证器检查的外部 artifact"，而这正是本周六篇论文从不同角度共同指向的同一个方向。
