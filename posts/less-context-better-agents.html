"Less Context, Better Agents" — 当少即是多：Context Pruning 如何让 Agent 性能跳升 20pp

July 5, 2026

给 AI Agent 更多的 context 信息，它就能表现得更好——这个直觉似乎不言自明。但一篇来自 Microsoft Dynamics 365 团队的 arXiv 论文用精确的实验数据挑战了这个假设：老 context 不仅是冗余的，而且是主动有害的。

Context Rot：当历史变成负担

企业 ERP 系统的 tool response 极其冗长。D365 F&O 的单次 MCP tool 调用返回的表单快照可达 500–3,000 tokens——包含完整的字段值、导航面包屑、系统元数据。当一个 Agent 执行 15–30 次 tool 调用才能完成一项费用逐项分解任务时，全量保留对话历史会导致三个问题：

  1. Context overflow：token 预算耗尽，任务中断
  2. Stale-state errors：Agent 基于早已过时的表单快照做决策——例如重复添加已经存在的 line item
  3. 推理成本线性增长：全量历史 1.48M tokens，而核心决策需要的仅是最新的工作状态

这种现象在生产系统中被称为 "context rot"——模型的有效召回能力在 token 数增长时退化，远早于硬 context 限制被触及（Anthropic, 2025）。

实验设计：四个配置，一个变量

论文评估了四种 context 管理策略，应用于 GPT-5 在 D365 F&O 上执行 50 任务酒店费用逐项分解 benchmark。每个配置运行 5 次取均值。

配置User ModelContext 策略目的
C1✗全量历史消融基线（GPT-5 无 user model 时会 stall）
C2✓全量历史Context-engineering 比较基线
C3✓仅保留最近 5 次 tool call纯 pruning
C4✓最近 5 次 + 摘要（窗口=3）Pruning + 轻量自动摘要

关键设计：C2→C3→C4 的比较中，user model 保持恒定（GPT-4.1）。论文明确声明 C1→C2 的跳跃不代表 context engineering 的价值——它反映的是 GPT-5 在无 user model 下的 stall 行为。真正的 context policy 效应仅在 C2→C3→C4 中体现。

  • C3→C4：完成率 +12.6pp，tokens 仅 +3.4%（摘要的边际成本极低）

这不是渐进改进——C4 在 性能 和 效率 两个维度上同时击败了全量历史基线。

交叉验证：Claude Sonnet 4.5

为了验证策略非模型特定，论文在 Sonnet 4.5 上复现了实验：

配置完全完成率
Full context（无 CE）88.0%
Pruning（Last 5）92.0%
Pruning + Summ.94.5%

Sonnet 不 stall（C1 即可达 88%），证明 GPT-5 的 C1 低性能是模型特异的。但方向一致：pruning + summary 在两个模型族上都优于全量历史。 时间溢价稳定在 ~5.6%。

为什么 Pruning 能击败全量历史？

答案在失败模式分析中。论文对每次非完成任务分类为 6 种失败模式（pooled over 5 runs）：

失败模式C2C3C4
Stale-state reference3464
Wrong subcategory mapping896
Duplicate / skipped repeat12115
Premature termination9183
Tool / form navigation error652
Residual amount mismatch441
Total non-completions735321

这张表揭示了一个清晰的机制链条：

  1. C2 → C3：stale-state 错误从 34 骤降至 6（pruning 消除了过时表单快照）。但 premature termination 从 9 暴增至 18——Agent 失去了对"我已经完成了多少"的全局感知。
  2. C3 → C4：premature termination 从 18 降至 3（摘要恢复了任务进度感知），同时 stale-state 错误保持低位（4）。

这是一种精巧的 信息分工：pruning 提供当前本地状态（刚刚做了什么、结果如何），摘要提供全局任务进度（已处理多少项、还剩多少金额）。两者组合构成完整的工作记忆，缺一不可。

论文进一步量化了这种分工的成本效益：摘要仅增加 3.4% 的 token 开销（C3→C4），换来了 12.6pp 的完成率提升。ROI 极高。

对 Agent 基础设施的工程启示

1. Context policy 应以 tool 交互为语义单元，而非 token 级别

这是论文与 token-level 压缩方法（LLMLingua, Selective Context）的本质区别。tokn-level 压缩可能破坏 ERP 表单所需的逐字控制名和数值。论文的 Algorithm 1 用 20 行伪代码实现了以完整 tool call/response pair 为粒度的 pruning，保证被保留的信息不残缺。

2. Inference-time policy 的价值远超训练期方案

不需要模型重训练、不需要外部记忆存储、不需要学习压缩器——一个固定窗口的 inference-time policy 就实现了 +20pp 完成率和 63% 成本削减。对于任何生产系统，这都是远比"等待下一个模型升级"务实的选择。

3. 固定窗口策略已足够好

敏感性分析显示从 N=5 到 N=10 的增益不到 1pp。这意味着系统不需要复杂的动态决策——固定的、手工选择的窗口已经捕获了绝大部分价值。工程复杂度大幅降低。

4. 摘要的 ROI 是世界级的

+3.4% tokens → +12.6pp 完成率。任何 Agent 系统都应该默认带一个 context summarizer。论文的摘要器是单次自由格式 LLM 调用（非结构化、非学习），成本极低，实现极其简单。

局限与未回答的问题

论文自述的边界：

  • 单一 ERP 工作流（费用管理），不代表全领域泛化
  • (N=5, W=3) 是手工选择，不是自适应的
  • 摘要器是单次 LLM 调用，未与结构化压缩器或 provider-native compaction API 对比
  • 未覆盖多 session 场景

值得关注的额外问题（基于方法论推断）：

问题潜在影响
Tool 响应格式单一（全部是 D365 F&O MCP 表单）若 tool 响应多样化（JSON + 自然语言 + 错误日志混合），摘要质量可能下降
对 reasoning-heavy 任务（代码生成、数学证明）是否有效？历史推理步骤可能有持续价值；pruning 可能损失关键中间推理，需要不同的窗口策略
窗口大小与任务结构的交互未建模N=5 基于领域知识（2–3 次 tool call 完成一个 line item）。如何自动选择 N？
未测试 adversarial 场景Pruning 窗口本身就是信息过滤——在安全敏感场景中是否可被利用？
摘要忠实度未量化论文仅给出 1 个定性样本（Appendix F）。摘要幻觉可能在不可察觉的情况下误导 Agent

一句话总结

Inference-time 的 recency pruning + 轻量摘要，不改模型、不加基础设施，让企业 ERP Agent 的完成率从 71% 跃至 91.6%，同时 token 成本降低 63%——old context is not just redundant, it's actively harmful.
