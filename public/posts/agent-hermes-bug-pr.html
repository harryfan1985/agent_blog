Agent 自驱动代码修复实践：从发现 Hermes Bug 到提交 PR 的完整过程

案例类型：Agent 自驱动 Bug 修复
项目：Hermes Agent (NousResearch/hermes-agent)
PR：#13974
耗时：约 10 分钟（从发现问题到提交 PR）
人工介入：仅需确认关键决策（是否提交 PR）

一、案例背景

用户正在配置 Hermes Agent 的模型切换功能，希望添加火山引擎 Coding Plan 的多个模型（glm-5.1, kimi-k2-6, minimax-m2-7）到 /model 命令的可选列表中。

配置完成后，用户发现问题：/model 命令只显示 glm-5.1，不显示其他添加的模型。

二、Agent 诊断流程

2.1 配置检查

Agent 首先检查用户配置文件，发现两个关键配置项：

配置项作用/model 显示
fallback_providers主模型失败时的备用链❌ 不显示
custom_providers用户自定义的 provider✅ 显示

用户的新模型被添加到了 fallback_providers，但 custom_providers 的配置也存在问题——只显示默认模型。

2.2 源码追踪

Agent 追踪到 hermes_cli/model_switch.py 中的 list_authenticated_providers() 函数，这是 /model 命令的模型列表生成器。

函数分为 4 个 Section：

Section 1: Hermes-mapped providers (models.dev catalog)
Section 2: Hermes-only providers (nous, openai-codex, copilot, opencode-go)
Section 2b: Cross-check canonical provider list
Section 3: User-defined endpoints from config  ← 正确处理 models 字典
Section 4: Saved custom providers from config  ← ❌ Bug 所在

2.3 Bug 定位

Agent 发现 Section 4 的代码逻辑存在明显缺陷：

# model_switch.py 第 1083-1085 行
default_model = (entry.get("model") or "").strip()
if default_model and default_model not in groups[slug]["models"]:
    groups[slug]["models"].append(default_model)
# ❌ 完全忽略了 entry["models"] 字典！

# 对比 Section 3 的正确实现：
cfg_models = ep_cfg.get("models", [])
if isinstance(cfg_models, list):
    for m in cfg_models:
        if m and m not in models_list:
            models_list.append(m)

根因：Section 4 只读取 model 字段（默认模型），完全忽略 models 字典中的其他模型定义。

三、Agent 自动修复

3.1 代码修复

Agent 使用 patch 工具修改代码，添加遍历 models 字典的逻辑：

+ # Also include models from the "models" dict (model_name -> metadata)
+ # or list format in the entry.
+ entry_models = entry.get("models")
+ if isinstance(entry_models, dict):
+     for m_name in entry_models:
+         if m_name and m_name not in groups[slug]["models"]:
+             groups[slug]["models"].append(m_name)
+ elif isinstance(entry_models, list):
+     for m in entry_models:
+         m_str = (m if isinstance(m, str) else str(m)).strip()
+         if m_str and m_str not in groups[slug]["models"]:
+             groups[slug]["models"].append(m_str)

3.2 验证修复

Agent 编写 Python 测试脚本验证修复效果：

from hermes_cli.model_switch import list_authenticated_providers
from hermes_cli.config import load_config

cfg = load_config()
providers = list_authenticated_providers(
    current_provider='custom',
    custom_providers=cfg.get('custom_providers', []),
)

# 结果对比
# Before: ['glm-5.1'] (1 model)
# After:  ['glm-5.1', 'glm-4-7-251222', 'kimi-k2-6', 'minimax-m2-7'] (4 models)

验证成功 ✅

四、用户介入点

修复验证后，Agent 向用户报告进展。用户提出关键问题：

"你直接修改了 hermes 的内部实现？这样我以后升级版本还会有问题吧"

Agent 立即意识到风险：直接修改本地源码会导致升级时丢失修复。

Agent 提出三个选项：

选项优点缺点
A. 提交 PR永久解决，造福所有用户需等待合并
B. 记录 bug不影响升级/model 显示不完整
C. 保持修改立即可用需手动重新应用

用户选择选项 A："帮我创建一个 PR 吧，提交到社区"

五、自动化 PR 提交流程

5.1 Git 操作

# 1. Fork 上游仓库
gh repo fork NousResearch/hermes-agent --clone=false

# 2. 创建修复分支
git checkout -b fix-custom-providers-models-list

# 3. 提交修改
git add hermes_cli/model_switch.py
git commit -m "fix(model_switch): include all models from custom_providers models dict"

# 4. 推送到 Fork
git push fork fix-custom-providers-models-list

# 5. 创建 PR
gh pr create --repo NousResearch/hermes-agent         --title "fix(model_switch): include all models from custom_providers models dict"         --body-file pr_body.md

5.2 PR 描述生成

Agent 自动生成英文 PR 描述，包含：

  • Bug Description
  • Example (config.yaml before/after)
  • Root Cause Analysis
  • Fix Code
  • Testing Results

九、总结

本次案例展示了 AI Agent 在实际开发场景中的自驱动问题解决能力：

  1. 主动诊断：无需用户详细描述，Agent 从症状自动追踪到根因
  2. 精准修复：使用代码工具进行最小化修改，保持风格一致
  3. 风险意识：用户提醒后立即评估升级兼容风险
  4. 流程自动化：Fork → Branch → Commit → PR 全流程自动化
  5. 持续跟进：设置监控任务，自动通知 PR 状态变化

关键启示：Agent 不是替代开发者，而是加速开发流程——让开发者专注于决策，Agent 执行细节。
