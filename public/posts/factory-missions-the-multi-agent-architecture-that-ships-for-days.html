Factory Missions: The Multi-Agent Architecture That Ships for Days

May 21, 2026

Overview

At AI Engineer London 2026, Factory's Luke Alvoeiro dropped a thesis that's been echoing through the AI infrastructure community: "The bottleneck in software engineering nowadays is not intelligence. It's now limited by human attention."

This article breaks down Factory's Missions architecture — a multi-agent system that ran a 16-day autonomous build of a Slack clone with 89.25% test coverage, requiring only periodic human check-ins. We dive into the engineering principles behind it, the production data, and what it means for anyone building agent infrastructure today.

The Core Problem: Context Toxicity

Factory's official architecture doc (by Theo Luan) frames the problem precisely: agents are highly reactive to their context. An agent's trajectory is append-only — every past thought, observation, and action feeds into its current reasoning. Models seek coherence, integrating everything they've seen into a unified worldview.

This creates two failure modes:

  • Context Dilution — As scope broadens, irrelevant information accumulates. Factory measured this: a focused task has 88% signal, while a broad task drops to only 38%. Two-thirds of the context becomes noise.
  • Self-Evaluation Bias — An agent that implements code cannot objectively evaluate it. Its prior reasoning creates a gravitational field that anchors subsequent evaluation toward confirmation rather than discovery.

Role Definitions (Stricter Than You Think)

RoleSingle GoalExplicitly Avoids
OrchestratorPlan, decompose, steer execution through all validation gatesAccumulating granular context — delegates all investigation and implementation
WorkerComplete a well-specified feature with clear success criteriaJudging its own work — final correctness is the Validator's call
ValidatorEvaluate completed work, surface bugs and gapsImplementing fixes — only reports issues back to Orchestrator

Critically: Neither validator has seen the code before. They come in fresh, making validation adversarial by design — the same reason humans do code review.

Test-Driven Development at Two Levels

This is the most underappreciated innovation in the architecture:

Micro-level (Worker): Each Worker writes tests before code, ensuring tests reflect intended behavior rather than implementation details. No "tests shaped by the code" problem.

Macro-level (Mission): The Orchestrator writes a Validation Contract — a set of behavioral assertions — before defining any features. The ordering constraint prevents the plan from contaminating the definition of correctness.

A Validation Contract looks like this:

### VAL-AUTH-001: Successful login
A user with valid credentials submits the login form
and is redirected to the dashboard.
Tool: agent-browser
Evidence: screenshot, network(POST /api/auth/login -> 200)

### VAL-CROSS-001: Auth gates pricing
A guest user sees "Sign in for pricing" on the catalog.
After logging in, real prices are shown.
Tool: agent-browser
Evidence: screenshot(guest-view), screenshot(authed-view)

These assertions are verified by User-Testing Validators — agents that spawn the actual application and interact with it through computer-use, filling forms, clicking buttons, and verifying functional flows end-to-end. As Luke noted: "Most of the missions wall clock time is actually spent here, waiting for real-world execution to occur instead of generating tokens."

Structured Handoffs: The Connective Tissue

When a Worker finishes a feature, it doesn't just say "I'm done." It fills out a structured handoff detailing:

  • What was completed and what was left undone
  • Every command run and its exit code
  • Issues discovered during implementation
  • Whether it followed the Orchestrator's defined procedures

This is why the system self-heals. Errors get caught at milestone boundaries. Corrective work gets scoped by the Orchestrator as fix features. The system pulls itself back on track — "not by hoping agents remember what happened, but by forcing them to write it down and then actually address issues."

Serial Execution (The Counterintuitive Choice)

Parallelism seems obvious: 10 agents = 10x throughput. Factory tried it. It failed.

  • Agents conflict on shared code
  • They duplicate work
  • They make inconsistent architectural decisions
  • Coordination overhead eats the speed gains while burning tokens

Missions runs features serially — only one Worker or Validator at any given time. Within a feature, read-only operations (code search, API research) are parallelized. Within validators, code review is parallelized across features.

"It seems slower on paper, but the error rate drops dramatically, and when you have tasks that run for many days, this sort of correctness compounds."

Production Data: Building a Slack Clone

Factory's official blog post provides exact metrics from a full Slack clone build:

pie title Time Allocation (16.5h total) "Implementation" : 9.98 "Validation" : 6.14 "Orchestration" : 0.38
  • 38,800 lines of code — 52.5% tests (20,400 test lines / 18,500 source lines)
  • 89.25% statement coverage
  • 778.5M total tokens (744.9M cache reads — prompt caching is critical for cost)
  • 185 total agent runs: 63 Workers, 27 Validators (with 82 sub-agents), 1 Orchestrator (12 sub-agents)
  • Median trajectory: 51 turns for implementation, 30 turns for validation

The Reliability Loop

0 out of 6 milestones passed on the first validation round. Every milestone needed 2-4 rounds to converge. Validators surfaced 81 issues (65 blocking, 11 non-blocking, 5 suggestions), which the Orchestrator turned into 21 fix features — 34.4% of the total implementation workload.

This number — one-third of all work being fixes — is both impressive and sobering. Impressive because the system catches and fixes these autonomously. Sobering because it reveals the inherent error rate of agent-generated code.

Droid Whispering: The New Engineering Skill

Luke introduced a concept he calls "Droid Whispering" — the ability to mentally model how different LLMs interact, where they fail, and how those failures compound over multi-day runs.

  • Planning → slow, careful reasoning → frontier reasoning models
  • Implementation → fast code fluency and creativity → cost-efficient coding models
  • Validation → precise instruction following → even a different provider to avoid same-training-data bias

"No single model nor model provider is best at all three of these. If you're locked into one model provider, you're constrained by that family's weakest capability."

The "Bitter Lesson" Safety Net

Every multi-agent builder fears the next model release making their architecture obsolete. Factory's defense: the orchestration logic is almost entirely in prompts and skills, not a hard-coded state machine.

"How it decomposes failures and handles failures is all in about 700 lines of text. Four sentences of this can alter the execution strategy pretty dramatically."

The only deterministic logic is thin bookkeeping — running validation, blocking progress when handoffs flag issues. The intelligence is in the prompts, meaning every model improvement automatically improves the entire system.

Five Communication Patterns Composed

Alvoeiro's taxonomy of multi-agent communication patterns, all composed within Missions:

  1. Delegation — Orchestrator spawns Workers and research sub-agents
  2. Creator-Verifier — Implementation and validation are always separate agents with separate context
  3. Broadcast — Shared mission state that every agent references
  4. Negotiation — At milestone boundaries, Orchestrator evaluates handoff summaries against the Validation Contract
  5. Direct Communication — Reserved for specialized cases; state fragmentation is the risk

"Strategies aren't enough. You need the connective tissue — structured handoffs, the right model in each role, and an architecture that improves with every model generation."

Implications for Agent Infrastructure

If you're building agent orchestration systems (Hermes, OpenCode, OmO, custom harnesses), here's what to steal from Factory's playbook:

  1. Validation Contract layer — Define "correct" before "how to build." Independent validators with fresh context. This is the single biggest missing piece in most agent systems today.
  2. Structured handoffs — Sub-agent results must include: what was done, what wasn't, commands run with exit codes, issues found. Free-text summaries are insufficient for multi-day autonomy.
  3. Model selection per role — Orchestrator uses pro (expensive reasoning), Workers use flash (cheap execution), Validators should use a different provider to eliminate training-data correlation bias.
  4. Serial feature execution — Parallelism in code modification creates more coordination cost than it saves. Parallelize read-only operations instead.
  5. Prompt-driven orchestration — Hard-coded state machines ossify. Prompt-driven logic improves with every model release. The "bitter lesson" of multi-agent systems.

The Bottom Line

Factory's Missions demonstrates that multi-day autonomous software development is not a research fantasy — it's an engineering reality today. The key insight isn't about better models. It's about better orchestration: how to split work, verify it, and keep agents from stepping on each other across days of continuous operation.

As models continue to improve, the competitive edge shifts from who has the smartest model to who has the smartest composition of models. The winners will be those who think in terms of agent ecosystems, not single-agent sessions.

"The people who develop an intuition for how different models compose under pressure — those folks are going to be really shipping the next generation of innovation."
