从论文到实践：面向意图对齐的 Coding Agent 工作流与公共机制设计

May 24, 2026

在分析完 Intent-Centric SE (arXiv:2605.11027) 和 Intent Formalization (arXiv:2603.17150) 两篇论文，并深入研究了 TiCoder 和 VeriStruct 的开源实现后，我设计了一套完整的 Coding Agent 工作流和公共机制。

阶段 1：意图澄清 — 歧义驱动的交互式形式化

借鉴 TiCoder 的核心机制：AI 生成代码的最大问题不是"写不出"，而是"写了但不对"。同一个 prompt 可以对应多种语义——"移除列表中的重复项"是保留一个副本还是删除所有重复？

工作机制

  1. LLM 生成 N 个候选实现（N≥3，温度>0，确保多样性）
  2. 自动找 Disagreement Points——不同候选产生不同输出的输入点
  3. 在分歧最大的 K 个点向用户提问（Yes/No/Expected Output）
  4. 用户反馈剪枝错误候选→ 批准的测试持久化为回归基线
  5. 输出 Formalized Intent：{已批准测试, 约束条件, 验收标准}

关键参数来自 TiCoder 的用户实验：N=5 即可有效覆盖歧义空间，K=3-5 保证认知负荷不超载（p=0.007）。提问策略用区分度排序（distinguishing power）——优先问能让候选聚类分得最开的测试点。

阶段 2：上下文策展 — 三层分层注入

不是把所有上下文塞进 prompt（token 爆炸），而是：

层级内容策略
Layer 1: 静态架构ADR、依赖图、接口契约、编码规范System prompt 注入
Layer 2: 动态仓库相关文件（AST调用链）、现有测试、最近变更BM25+向量检索 → LLM 重排序 → Tool output
Layer 3: 运行时错误日志、性能 profile、安全扫描On-demand 检索

阶段 3：生成-验证-修复循环 — Planner + Repair Router

这是整个工作流的核心引擎，借鉴了 VeriStruct（TACAS 2026）的架构：

Planner 状态机

状态触发动作
PLAN新任务开始分析任务，生成执行计划
GENERATE需要产出LLM 调用，注入上下文+失败历史
VERIFY生成完成测试+静态检查+规约验证
REPAIR验证失败错误分类→路由修复模块
DONE全部通过输出+证据包
STUCK循环>N次降级人工介入

Repair Router — 14+ 修复模块

Agent 生成代码后验证失败，如果没有错误分类，Agent 会盲目重试。解决方案是一个错误类型→修复模块的路由系统：


Error Classifier（错误分类器）
    ↓ 结构化错误类型
Repair Router（修复路由器）
    ↓ 匹配修复模块
├── repair_type        → 类型错误
├── repair_syntax      → 语法错误
├── repair_precond     → 前提条件太弱
├── repair_postcond    → 后置条件太强
├── repair_assertion   → 断言失败
├── repair_invariant   → 不变量不成立
├── repair_arithmetic  → 算术溢出
├── repair_missing     → 缺少导入/定义
└── ...
+ Loop Detector（收敛检测+超时保护）

阶段 4：证据门禁 — 六层自动化检查

AI 生成代码的速度远超人类审查速度。传统"逐行审查"无法跟上。需要多层自动化门禁：


Gate 1: 语法与类型   → 编译通过？类型检查通过？
Gate 2: 静态安全     → SAST 扫描？依赖漏洞？密钥泄露？
Gate 3: 意图符合性   → 通过所有批准测试？规约评分？
Gate 4: 回归保护     → 现有测试全部通过？
Gate 5: 架构一致性   → 违反架构约束？循环依赖？
Gate 6: 人类审查     → 按风险分级（低→自动合入，中→轻量审查，高→完整审查）

每次变更附带证据包（Evidence Package）：原始 prompt、批准测试、候选考虑数、选择理由、测试通过数、静态检查结果、规约 soundness/completeness 评分、diff 摘要、风险等级。

阶段 5：治理与审计

三大体系确保长期可追溯：

  • Agent 身份体系 — 每个 agent 有唯一 ID + 权限绑定 + 操作审计日志 + 会话溯源
  • Provenance Chain — NL Prompt → Intent → Plan → Tool Calls → Code → Gates → Human Decision → Merge，每环节有 timestamp/agent ID/input/output/rationale
  • Policy-as-Code — 基于 OPA/Rego，定义 agent 能改什么文件、能调什么工具、什么变更需审批、什么时间窗口允许部署

两篇论文的互补对话

设计这套工作流的过程本身验证了两篇论文的核心论点：

维度De La Cruz (2605.11027)Lahiri (2603.17150)
方法定性话语分析技术立场论文
视角"软件工程职业如何转型""代码可靠性如何保证"
回答意图中心 + 治理 + 问责意图形式化光谱 + 自动化指标
交集在 vibe coding 时代，人类可能永远不检查代码——规约成为人机之间的主要界面

De La Cruz 告诉我们为什么软件工程在转型（从 code-centric 到 intent-centric），Lahiri 告诉我们转型中怎么做才能保证可靠（从测试到 DSL 的形式化光谱）。

实施优先级

优先级做什么理由
P0 立即错误分类器 + Repair Router最小改动最大收益
P1 近期Disagreement Detector借鉴 TiCoder 的 test_ranking.py
P2 近期Evidence Package + Gate Pipeline结构化审批流程
P3 中期Spec Soundness/Completeness需要 nl2postcond 的规约评估逻辑
P4 远期Provenance Tracker + Policy Engine当 agent>3 且做生产变更时
