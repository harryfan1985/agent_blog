# Hermes Agent 技术报告 **项目名称**: Hermes Agent **组织**: Nous Research **仓库**: https://github.com/NousResearch/hermes-agent **文档**: https://hermes-agent.nousresearch.com/docs/ **许可证**: MIT **分析日期**: 2026-04-11 --- ## 执行摘要 Hermes Agent 是 Nous Research 开发的自改进 AI 助手框架，核心特点包括内置学习循环、多平台支持、多模型兼容性以及强大的长期记忆和自演进机制。该框架可在 $5 VPS、GPU 集群或无服务器基础设施上运行，支持 CLI、Telegram、Discord、Slack、WhatsApp、Signal 等多种交互方式。 本报告详细分析了 Hermes Agent 的整体架构、核心子系统、长期记忆机制、自演进能力以及设计哲学。 --- ## 一、项目概述 ### 1.1 核心特性 - **自改进能力**: 内置学习循环，从经验中创建技能并持续改进 - **多平台支持**: CLI、Telegram、Discord、Slack、WhatsApp、Signal、Email 等 - **多模型兼容**: 支持 OpenRouter、Nous Portal、OpenAI、Anthropic、z.ai/GLM、Kimi/Moonshot、MiniMax 等 - **灵活部署**: 可在 $5 VPS、GPU 集群或无服务器基础设施上运行 - **完整 TUI**: 全功能终端界面，支持多行编辑、斜杠命令自动补全、对话历史、中断重定向、流式工具输出 - **跨平台会话连续性**: 语音备忘录转录、跨平台对话连续性 - **定时自动化**: 内置 cron 调度器，支持自然语言任务调度 - **并行委派**: 可生成隔离子代理进行并行工作流 - **研究就绪**: 批量轨迹生成、Atropos RL 环境、轨迹压缩 ### 1.2 安装与快速开始 ```bash curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash source ~/.bashrc hermes ``` 支持平台：Linux、macOS、WSL2、Android (Termux) --- ## 二、核心架构 ### 2.1 系统架构图 ``` (~9,200 行) **核心职责**: - 组装有效系统提示词和工具架构 - 选择正确的 Provider/API 模式 - 执行可中断的模型调用 - 执行工具调用（顺序或并发） - 维护对话历史（OpenAI 消息格式） - 处理压缩、重试、回退模型切换 - 跟踪父代理和子代理的迭代预算 - 在上下文丢失前刷新持久记忆 #### API 模式 | 模式 | 用途 | 客户端类型 | |------|------|-----------| | `chat_completions` | OpenAI 兼容端点 | openai.OpenAI | | `codex_responses` | OpenAI Codex/Responses API | openai.OpenAI | | `anthropic_messages` | Anthropic 原生 API | anthropic.Anthropic | **模式解析顺序**: 1. 显式 `api_mode` 构造函数参数（最高优先级） 2. Provider 特定检测（如 anthropic → anthropic_messages） 3. Base URL 启发式（如 api.anthropic.com → anthropic_messages） 4. 默认: chat_completions #### 轮次生命周期 ``` run_conversation() 1. 生成 task_id（如未提供） 2. 追ser 用户消息到对话历史 3. 构建或重用缓存的系统提示词 4. 检查是否需要预压缩（>50% 上下文） 5. 从对话历史构建 API 消息 6. 注入临时提示词层（预算警告、上下文压力） 7. 应用提示词缓存标记（如在 Anthropic） 8. 执行可中断 API 调用 9. 解析响应: - 如 tool_calls: 执行它们，追加结果，循环回步骤 5 - 如文本响应: 持久化会话，如需要刷新记忆，返回 ``` #### 消息格式 所有消息使用 OpenAI 兼容格式： ```json {"role": "system", "content": "..."} {"role": "user", "content": "..."} {"role": "assistant", "content": "...", "tool_calls": [...]} {"role": "tool", "tool_call_id": "...", "content": "..."} ``` **推理内容**: 存储在 `assistant_msg["reasoning"]`，可选通过 `reasoning_callback` 显示。 #### 消息交替规则 Agent 循环强制严格的消息角色交替： - 系统消息后: User → Assistant → User → Assistant → ... - 工具调用期间: Assistant (with tool_calls) → Tool → Tool → ... → Assistant - 永不两个 assistant 消息连续 - 永不两个 user 消息连续 - 只有 tool 角色可以有连续条目（并行工具结果） #### 可中断 API 调用 ``` ┌──────────────────────┐ ┌──────────────┐ │ Main thread │ │ API thread │ │ wait on: │────▶│ HTTP POST │ │ - response ready │ │ to provider │ │ - interrupt event │ └──────────────┘ │ - timeout │ └──────────────────────┘ ``` 当被中断（用户发送新消息、/stop 命令或信号）： - API 线程被放弃（响应丢弃） - Agent 可以处理新输入或干净关闭 - 不会将部分响应注入对话历史 #### 工具执行 **顺序 vs 并发**: - 单个工具调用 → 直接在主线程执行 - 多个工具调用 → 通过通过 ThreadPoolExecutor 并发执行 例外：标记为交互式的工具（如 clarify）强制顺序执行。 **执行流程**: ``` for each tool_call in response.tool_calls: 1. 从 tools/registry.py 解析处理器 2. 触发 pre_tool_call 插件 hook 3. 检查是否危验命令 (tools/approval.py) - 如危验: 调用 approval_callback，等待用户 4. 用 args + task_id 执行处理器 5. 触发 post_tool_call 插件 hook 6. 追ser {"role": "tool", "content": result} 到历史 ``` #### Agent 级工具 有些工具在到达 `handle_function_call()` 前被 `run_agent.py` 拦截： | 工具 | 拦截原因 | |------|----------| | todo | 读写 agent 本地任务状态 | | memory | 写入持久记忆文件 | | session_search | 查询 agent 的会话 DB | | delegate_task | 生成子代理会话 | 这些工具直接修改 agent 状态并返回合成工具结果，不通过注册表。 #### 回调表面 | 回调 | 触发时机 | 使用者 | |------|----------|---------| | tool_progress_callback | 工具执行前后 | CLI spinner、网关进度消息 | | thinking_callback | 模型开始/停止思考 | CLI "thinking..." 指示器 | | reasoning_callback | 模型返回推理内容 | CLI 推理显示、网关推理块 | | clarify_callback | clarify 工具被调用 | CLI 输入提示、网关交互消息 | | step_callback | 每个完整 agent 轮次后 | 网关步骤跟踪、ACP 进度 | | stream_delta_callback | 每个流式 token（启用时） | CLI 流式显示 | | tool_gen_callback | 从流解析工具调用 | CLI spinner 中工具预览 | | status_callback | 状态变化 | ACP 状态更新 | #### 预算和回退行为 **迭代预算**: - 默认: 90 轮次（可通过 `agent.max_turns` 配置） - 在父代理和子代理间共享 - 两层预算压力: - 70%+ 使用: 追ser `[BUDGET: Iteration X/Y. N iterations left. Start consolidating your work.]` - 90%+ 使用: 追ser `[BUDGET WARNING: Iteration X/Y. Only N iteration(s) left. Provide your final response NOW.]` - 100%: Agent 停止并返回工作摘要 **回退模型**: 当主模型失败（429 速率限制、5xx 服务器错误、401/403 认证错误）： - 检查配置中的 `fallback_providers` 列表 - 按顺序尝试每个回退 - 成功后用新 provider 继续对话 - 401/403 时，失败前尝试凭证刷新 ### 3.2 Prompt 系统 **文件**: `agent/prompt_builder.py` **系统提示词组装来源**: 1. **个性** (SOUL.md) - Agent 的人格和语调 2. **记忆** (MEMORY.md, USER.md) - 长期记忆和用户画像 3. **技能** - 可用技能列表和内容 4. **上下文文件** (AGENTS.md, .hermes.md) - 项目上下文 5. **工具使用指导** - 工具调用指南 6. **模型特定指令** - 针对特定模型的指令 **提示词缓存** (`agent/prompt_caching.py`): - 应用 Anthropic 缓存断点 - 支持 5m 或 1h TTL - 降低多轮对话输入 Token 成本 ~75% **上下文压缩** (`agent/context_compressor.py`): - 默认压缩引擎 - 有损摘要算法 - 保护最后 N 条消息（默认 20） ### 3.3 Provider 解析 **文件**: `hermes_cli/runtime_provider.py` **功能**: - 共享运行时解析器，被 CLI、网关、cron、ACP 和辅助调用使用 - 映射 `(provider, model)` 元组到 `(api_mode, api_key, base_url)` - 处理 18+ Provider - OAuth 流程处理 - 凭证池管理 - 别名解析 **支持的 Provider**: - OpenRouter, Nous Portal, OpenAI, Anthropic - z.ai/GLM, Kimi/Moonshot, MiniMax - Groq, Together, xAI, DeepInfra - 以及更多... ### 3.4 工具系统 **文件**: `tools/registry.py` **统计**: 47 个工具，20 个工具集 **工具注册机制**: ```python registry.register( name="terminal", toolset="terminal", schema={...}, # OpenAI function-calling schema handler=handle_terminal, check_fn=check_terminal, # 可用性检查 requires_env=["SOME_VAR"], is_async=False, description="Run commands", emoji="💻" ) ``` **工具可用性检查** (`check_fn`): - API Key 存在性检查 - 服务运行状态检查 - 二进制安装检查 **工具集解析**: - 显式启用/禁用工具集列表 - 平台预设 (hermes-cli, hermes-telegram 等) - 动态 MCP 工具集 - 特殊用途集如 hermes-acp **工具分发流程**: ``` Model response with tool_call ↓ run_agent.py agent loop ↓ model_tools.handle_function_call(name, args, task_id, user_task) ↓ [Agent-loop tools?] → 直接由 agent 循环处理 ↓ [Plugin pre-hook] → invoke_hook("pre_tool_call", ...) ↓ registry.dispatch(name, args, **kwargs) ↓ 查找 ToolEntry by name ↓ [Async handler?] → 通过 _run_async() 桥接 [Sync handler?] → 直接调用 ↓ 返回结果字符串（或 JSON 错误） ↓ [Plugin post-hook] → invoke_hook("post_tool_call", ...) ``` **危险命令批准流程** (`tools/approval.py`): **检测模式**: - 递归删除 (`rm -rf`) - 文件系统格式化 (`mkfs`, `dd`) - SQL 破坏性操作 - 系统配置覆盖 - 服务操作 - 远程代码执行 - Fork bombs、进程 kills 等 **批准流程**: 1. 模式检测 2. 如匹配: - CLI 模式: 交互式提示 - 网关模式: 异步批准回调 3. 会话级跟踪 4. 永久 allowlist 选项 **终端后端** (6 种): - local, Docker, SSH, Daytona, Modal, Singularity ### 3.5 会话持久化 **文件**: `hermes_state.py` **特性**: - SQLite + FTS5 全文搜索 - 会话谱系追踪（父/子关系） - 平台隔离 - 原子写入 + 竞争处理 **数据流**: ``` 每轮对话后 ↓ 消息保存到会话存储 ↓ 记忆变更刷新到 MEMORY.md / USER.md ↓ 会话可通过 /resume 或 hermes chat --resume 恢复 ``` ### 3.6 消息网关 **文件**: `gateway/run.py` (~7,500 行) **15 个平台适配器**: - Telegram, Discord, Slack, WhatsApp, Signal - Matrix, Mattermost, Email, SMS - DingTalk, Feishu, WeCom, Weixin, BlueBubbles - Home Assistant, Webhook **功能**: - 统一会话路由 - 用户授权（allowlist + DM 配对） - 斜杠命令分发 - Hook 系统 - Cron 计时 - 后台维护 **数据流**: ``` 平台事件 → Adapter.on_message() → MessageEvent → GatewayRunner._handle_message() → 授权用户 → 解析会话 key → 创建 AIAgent + 会话历史 → AIAgent.run_conversation() → 通过适配器投递响应 ``` ### 3.7 插件系统 **发现来源**: 1. `~/.hermes/plugins/` (用户) 2. `.hermes/plugins/` (项目) 3. pip entry points **插件类型**: - 工具插件 - Hook 插件 - CLI 命令插件 - Memory Provider 插件（单选） - Context Engine 插件（单选） ### 3.8 Cron 调度器 **特性**: - JSON 任务存储 - 多种调度格式 - 可附加技能和脚本 - 投递到任意平台 **数据流**: ``` 调度器 tick → 从 jobs.json 加载到期任务 → 创建新的 AIAgent（无历史） → 注入附加技能作为上下文 → 运行任务提示词 → 投递到目标平台 → 更新任务状态和 next_run ``` ### 3.9 ACP 集成 **文件**: `acp_adapter/` **功能**: - 通过 stdio/JSON-RPC 暴露给编辑器 - 支持 VS Code, Zed, JetBrains ### 3.10 RL/环境/轨迹 **特性**: - Atropos 集成 - 多工具调用解析器 - ShareGPT 格式轨迹生成 --- ## 四、长期记忆系统 ### 4.1 内置记忆（核心层） Hermes 采用**双层记忆架构**，在会话开始时注入系统提示词： | 记忆文件 | 用途 | 字符限制 | Token 估算 | |----------|------|----------|-----------| | `MEMORY.md` | Agent 个人笔记（环境事实、约定、教训） | 2,200 字符 | ~800 tokens | | `USER.md` | 用户画像（偏好、沟通风格、期望） | 1,375 字符 | ~500 tokens | **特点**: - **冻结快照模式**: 在会话开始时一次性注入，会话中不改变 - **Agent 自主管理**: 通过 `memory` 工具自动 add/replace/remove - **字符限制**: 保持记忆聚焦，强制精集 - **安全扫描**: 检测提示注入、凭证泄露、SSH 后门等威胁 **记忆注入格式**: ``` ══════════════════════════════════════════════ MEMORY (your personal notes) [67% — 1,474/2,200 chars] ══════════════════════════════════════════════ User's project is a Rust web service at ~/code/myapi using Axum + SQLx § This machine runs Ubuntu 22.04, has Docker and Podman installed § User prefers concise responses, dislikes verbose explanations ``` **记忆工具操作**: - `add` - 添加新记忆条目 - `replace` - 替换现有条目（使用子串匹配） - `remove` - 移除不再相关的条目（使用子串匹配） **子串匹配**: `replace` 和 `remove` 使用短唯一子串匹配： ```python memory(action="replace", target="memory", old_text="dark mode", content="User prefers light mode in VS Code, dark mode in terminal") ``` ### 4.2 外部 Memory Providers（扩展层） Hermes 集成了 **8 个外部记忆提供商**，提供超越内置记忆的深度知识管理： | Provider | 存储方式 | 成本 | 独特能力 | |----------|----------|------|----------| | **Honcho** | Cloud/自托管 | 付费/免费 | 辩证式用户建模、语义搜索 | | **OpenViking** | 自托管 | 免费 | 文件系统层级、分层加载 | | **Mem0** | Cloud | 付费 | 服务端 LLM 自动提取 | | **Hindsight** | Cloud/本地 | 付费/免费 | 知识图谱、实体解析、跨会话语义合成 | | **Holographic** | 本地 SQLite | 免费 | HRR 代数查询、信任评分 | | **RetainDB** | Cloud | $20/月 | 混合搜索、7 种记忆类型 | | **ByteRover** | 本地/Cloud | 免费/付费 | 预压缩提取、知识树 | | **Supermemory** | Cloud | 付费 | 上下文隔离、会话图谱构建 | #### Honcho **最佳用途**: 多代理系统与跨会话上下文、用户-代理对齐 **要求**: `pip install honcho-ai` + API Key 或自托管实例 **工具**: - `honcho_profile` (peer card) - `honcho_search` (语义搜索) - `honcho_context` (LLM 合成) - `honcho_conclude` (存储事实) **关键配置**: ```json { "apiKey": "your-key-from-app.honcho.dev", "hosts": { "hermes": { "enabled": true, "aiPeer": "hermes", "peerName": "your-name", "workspace": "hermes" } } } ``` **多代理/Profiles**: 每个 Hermes profile 获得自己的 Honcho AI peer，同时共享同一 workspace。 #### OpenViking **最佳用途**: 自托管知识管理与结构化浏览 **要求**: `pip install openviking` + 运行服务器 **工具**: - `viking_search` (语义搜索) - `viking_read` (分层: abstract/overview/full) - `viking_browse` (文件系统导航) - `viking_remember` (存储事实) - `viking_add_resource` (摄取 URLs/docs) **关键特性**: - 分层上下文加载: L0 (~100 tokens) → L1 (~2k) → L2 (full) - 具体文件路径、命令、结果] ### In Progress [当前进行的工作] ### Blocked [阻塞或遇到的问题] ## Key Decisions [重要技术决策及原因] ## Relevant Files [读取、修改或创建的文件 - 简短说明] ## Next Steps [接下来需要做什么] ## Critical Context [具体值、错误消息、配置详情] ``` **摘要预算**根据压缩的内容量缩放： - 公式: `content_tokens ×` 0.20（`_SUMMARY_RATIO` 常量） - 最小: 2,000 tokens - 最大: `min(context_length × 0.05, 12,000)` tokens #### Phase 4: 组装压缩消息 压缩消息列表为： - 头部消息（首次压缩时在系统提示词附加注释） - 摘要消息（角色选择以避免连续同角色违规） - 尾部消息（未修改） 孤立的 tool_call/tool_result 对通过 `_sanitize_tool_pairs()` 清理： - 引用已移除调用的工具结果 → 移除 - 结果被移除的工具调用 → 注入存根结果 ### 6.3 迭代式重新压缩 在后续压缩中，前摘要传递给 LLM 并指示更新而非从头摘要。这跨多次压缩保留信息——项目从 "In Progress" 移到 "Done"，添加新进度，移除过时信息。 压缩器实例上的 `_previous_summary` 字段为此目的存储上次摘要文本。 ### 6.4 压缩前后示例 #### 压缩前（45 条消息，~95K tokens） ``` [0] system: "YouYou are a helpful assistant..." (系统提示词) [1] user: "Help me set up a FastAPI project" [2] assistant: → terminal: mkdir project [3] tool: "directory created" [4] assistant: → write_file: main.py [5] tool: "file written (2.3KB)" ... 30 more turns of file editing, testing, debugging ... [38] assistant: → terminal: pytest [39] tool: "8 passed, 2 failed\n..." (5KB output) [40] user: "Fix failing tests" [41] assistant: → read_file: tests/test_api.py [42] tool: "import pytest\n..." (3KB) [43] assistant: "I see issue with the test fixtures..." [44] user: "Great, also add error handling" ``` #### 压缩后（25 条消息，~45K tokens） ``` [0] system: "You are a helpful assistant... [Note: Some earlier conversation turns have been compacted...]" [1] user: "Help me set up a FastAPI project" [2] assistant: "[CONTEXT COMPACTION] Earlier turns were compacted... ## Goal Set up a FastAPI project with tests and error handling ## Progress ### Done - Created project structure: main.py, tests/, requirements.txt - Implemented 5 API endpoints in main.py - Wrote 10 test cases in tests/test_api.py - 8/10 tests passing ### In Progress - Fixing 2 failing tests (test_create_user, test_delete_user) ## Relevant Files - main.py — FastAPI app with 5 endpoints - tests/test_api.py — 10 test cases - requirements.txt — fastapi, pytest, httpx ## Next Steps - Fix failing test fixtures - Add error handling" [3] user: "Fix failing tests" [4] assistant: → read_file: tests/test_api.py [5] tool: "import pytest\n..." [6] assistant: "I see issue with the test fixtures..." [7] user: "Great, also add error handling" ``` ### 6.5 Anthropic 提示词缓存 **策略**: system_and_3 Anthropic 允许每个请求最多 4 个 cache_control 断点。Hermes 使用 "system_and_3" 策略： ``` Breakpoint 1: 系统提示词（所有轮次稳定） Breakpoint 2: 倒数第 3 条非系统消息 ─┐ Breakpoint 3: 倒数第 2 条非系统消息 ├─ 滚动窗口 Breakpoint 4: 最后一条非系统消息 ─┘ ``` **工作原理**: `apply_anthropic_cache_control()` 深拷消息并注入缓存标记： ```python # 缓存标记格式 marker = {"type": "ephemeral"} # 或 1 小时 TTL: marker = {"type": "ephemeral", "ttl": "1h"} ``` 标记根据内容类型不同应用： | 内容类型 | 标记位置 | |----------|----------| | String content | 转换为 `[{"type": "text", "text": ..., "cache_control": ...}]` | | List content | 添加到最后元素的 dict | | None/empty | 添加为 `msg["cache_control"]` | | Tool messages | 添加为 `msg["cache_control"]`（仅原生 Anthropic） | **缓存感知设计模式**: - ✅ 稳定系统提示词：系统提示词是断点 1 并跨所有轮次缓存。避免对话中途变异（压缩仅在首次压缩时附加注释） - ✅ 消息顺序很重要：缓存命中需要前缀匹配。在中间添加或移除消息会使之后的所有内容缓存失效 - ✅ 压缩缓存交互：压缩后，压缩区域的缓存失效，但系统提示词缓存存活。滚动的 3 消息窗口在 1-2 轮内重建缓存 - ✅ TTL 选择：默认 5m（5 分钟）。对长运行会话使用 1h，其中用户在轮次之间休息 **启用提示词缓存**: 提示词缓存在以下情况自动启用： - 模型是 Anthropic Claude 模型（通过模型名检测） - Provider 支持 cache_control（原生 Anthropic API 或 OpenRouter） ```yaml # config.yaml — TTL 可配置 model: cache_ttl: "5m" # "5m" 或 "1h" ``` CLI 在启动时显示缓存状态： ``` 💾 Prompt caching: ENABLED (Claude via OpenRouter, 5m TTL) ``` ### 6.6 上下文压力警告 Agent 在压缩阈值的 85%（不是 85% 上下文——85% 的阈值为 50% 上下文）时发出上下文压力警告： ``` ⚠️ Context is 85% to compaction threshold (42,500/50,000 tokens) ``` 压缩后，如果使用率降到阈值的 85% 以下，警告状态被清除。如果压缩未能将使用率降到警告水平以下（对话太密集），警告持续，但压缩不会在再次超过阈值前重新触发。 --- ## 七、学习循环闭环 ### 7.1 完整学习流程 ``` 用户请求 ↓ Agent 执行任务（工具调用、推理） ↓ 成功完成？ ├─ 是 → 复杂任务？（5+ 工具调用） │ ├─ 是 → 提取工作流 → 创建技能 │ └─ 否 → 记录到记忆 │ └─ 否 → 遇到错误？ ├─ 是 → 找到解决路径？ │ ├─ 是 → 记录教训 → 更新技能 │ └─ 否 → 寻求帮助 │ └─ 否 → 用户纠正？ └─ 是 → 记录纠正 → 更新技能 ↓ 会话结束 ↓ 压缩上下文（保留关键信息） ↓ 提取记忆到外部 Provider ↓ 同步到会话数据库（FTS5 索引） ``` ### 7.2 记忆与技能的协同 | 层面 | 存储形式 | 用途 | 更新频率 | |------|----------|------|----------| | **短期记忆** | 对话历史 | 当前会话上下文 | 每轮 | | **程序性记忆** | 技能（SKILL.md） | 可工作流 | 按需创建 | | **长期记忆** | MEMORY.md/USER.md | 关键事实、用户画像 | 会话中 | | **外部记忆** | Provider 存储 | 深度知识图谱 | 每轮/会话结束 | | **历史会话** | SQLite + FTS5 | 跨会话回忆 | 自动 | ### 7.3 自演进的关键特性 **1. 主动记忆管理** - LLM 摘要提取关键信息 - 外部 Provider 提供语义搜索 **5. 多层记忆架构** - 内置记忆提供即时访问 - 外部 Provider 提供深度知识 - 会话搜索连接历史对话 --- ## 八、设计原则 | 原则 | 实践 | |------|------| | **提示词稳定性** | 系统提示词不会在对话中途改变。除非显式用户操作（/model），否则无缓存破坏变异 | | **可观察执行** | 每个工具调用对用户可见（通过回调）。CLI 中的进度更新（spinner）和网关中的聊天消息 | | **可中断** | API 调用和工具执行可中途取消（通过用户输入或信号） | | **平台无关核心** | 一个 AIAgent 类服务 CLI、网关、ACP、批量和 API 服务器。平台差异存在于入口点，而非 agent | | **松耦合** | 可选子系统（MCP、插件、memory providers、RL 环境）使用注册模式和 check_fn 门控，而非硬依赖 | | **Profile 配置隔离** | 每个 profile（`hermes -p `）获得自己的 HERMES_HOME、配置、记忆、会话和网关 PID。多个 profile 可并发运行 | --- ## 九、数据流详解 ### 9.1 CLI 会话流程 ``` 用户输入 → HermesCLI.process_input() → AIAgent.run_conversation() → prompt_builder.build_system_prompt() → runtime_provider.resolve_runtime_provider() → API 调用 → tool_calls? → model_tools.handle_function_call() → 循环 → 最终响应 → 显示 → 保存到 SessionDB ``` ### 9.2 网关消息流程 ``` 平台事件 → Adapter.on_message() → MessageEvent → GatewayRunner._handle_message() → 授权用户 → 解析会话 key → 创建 AIAgent + 会话历史 → AIAgent.run_conversation() → 通过适配器投递响应 ``` ### 9.3 Cron 任务流程 ``` 调度器 tick → 从 jobs.json 加载到期任务 → 创建新的 AIAgent（无历史） → 注入附加技能作为上下文 → 运行任务提示词 → 投递到目标平台 → 更新任务状态和 next_run ``` --- ## 十、安全特性 ### 10.1 危险命令检测 **检测模式** (`tools/approval.py`): | 类别 | 模式示例 | |------|----------| | 递归删除 | `rm -rf` | | 文件系统格式化 | `mkfs`, `dd` | | SQL 破坏性操作 | `DROP TABLE`, `DELETE FROM` without WHERE | | 系统配置覆盖 | `> /etc/` | | 服务操作 | `systemctl stop` | | 远程代码执行 | `curl \| sh` | | Fork bombs, 进程 kills | `:(){ :|:& };:`, `kill -9` | ### 10.2 命令批准流程 1. **模式检测**: `detect_dangerous_command(command)` 检查所有模式 2. **批准提示**: 如匹配： - CLI 模式: 交互式提示要求用户批准、拒绝或永久允许 - 网关模式: 异步批准回调发送请求到消息平台 3. **智能批准**: 可选地，辅助 LLM 可自动批准匹配模式的低风险命令（如 `rm -rf node_modules/` 安全但匹配 "recursive delete"） 4. **会话状态**: 批准按会话跟踪。一旦你为会话批准 "recursive delete"，后续 `rm -rf` 命令不再提示 5. **永久 allowlist**: "永久允许" 选项将模式写入 `config.yaml` 的 `command_allowlist`，跨会话持久 ### 10.3 记忆安全扫描 记忆条目在注入系统提示词前被扫描，因为它们被注入到系统提示词中。匹配威胁模式（提示注入、凭证泄露、SSH 后门）或包含不可见 Unicode 字符的内容被阻止。 --- ## 十一、与 OpenClaw 的对比 | 特性 | OpenClaw | Hermes Agent | |------|----------|---------------| | **内置记忆** | MEMORY.md | MEMORY.md + USER.md | | **外部 Provider** | 无 | 8 个集成 | | **技能系统** | 基础 | 完整（创建/更新/删除） | | **Skills Hub** | ClawHub | 7+ 源集成 | | **上下文压缩** | 手动 | 自动双层 | | **提示词缓存**` | 无 | Anthropic 缓存 | | **会话搜索** | 无 | FTS5 + LLM | | **自演进** | 有限 | 完整学习循环 | | **Provider 支持** | 有限 | 18+ | | **平台支持** | Telegram 等 | 15+ 平台 | | **RL/训练** | 无 | 完整框架 | --- ## 十二、代码规模 | 组件 | 行数 | |------|------| | `run_agent.py` | ~9,200 | | `cli.py` | ~8,500 | | `hermes_cli/main.py` | ~5,500 | | `gateway/run.py` | ~7,500 | | `hermes_cli/setup/setup.py` | ~3,100 | | `tools/mcp_tool.py` | ~2,200 | | **测试** | ~3,000+ | --- ## 十三、总结 Hermes Agent 是一个设计精良、功能完备的 AI 助手框架，特别适合需要长期记忆、多平台集成和工具调用能力的场景。 ### 核心优势 1. **真正的自演进能力**: 通过技能系统和记忆管理，Agent 能够从经验中学习并持续改进 2. **多层记忆架构**: 内置记忆 + 8 个外部 Provider，提供从即时访问到深度知识图谱的完整谱系 3. **智能上下文管理**: 双层压缩 + Anthropic 缓存，优化 Token 使用和成本 4. **平台无关核心**: 单一 AIAgent 类服务所有入口点，易于扩展和维护 5. **安全优先**: 危险命令检测、批准流程、记忆安全扫描等多层防护 6. **灵活部署**: 从 $5 VPS 到 GPU 集群，适应各种使用场景 ### 适用场景 -
