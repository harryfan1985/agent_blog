执行摘要

Hermes Agent 是 Nous Research 开发的自改进 AI 助手框架，核心特点包括内置学习循环、多平台支持、多模型兼容性以及强大的长期记忆和自演进机制。该框架可在 $5 VPS、GPU 集群或无服务器基础设施上运行，支持 CLI、Telegram、Discord、Slack、WhatsApp、Signal 等多种交互方式。

本报告详细分析了 Hermes Agent 的整体架构、核心子系统、长期记忆机制、自演进能力以及设计哲学。

一、项目概述

1.1 核心特性

  • 自改进能力: 内置学习循环，从经验中创建技能并持续改进
  • 多平台支持: CLI、Telegram、Discord、Slack、WhatsApp、Signal、Email 等
  • 多模型兼容: 支持 OpenRouter、Nous Portal、OpenAI、Anthropic、z.ai/GLM、Kimi/Moonshot、MiniMax 等
  • 灵活部署: 可在 $5 VPS、GPU 集群或无服务器基础设施上运行
  • 完整 TUI: 全功能终端界面，支持多行编辑、斜杠命令自动补全、对话历史、中断重定向、流式工具输出
  • 跨平台会话连续性: 语音备忘录转录、跨平台对话连续性
  • 定时自动化: 内置 cron 调度器，支持自然语言任务调度
  • 并行委派: 可生成隔离子代理进行并行工作流
  • 研究就绪: 批量轨迹生成、Atropos RL 环境、轨迹压缩

1.2 安装与快速开始

curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
source ~/.bashrc
hermes

支持平台：Linux、macOS、WSL2、Android (Termux)

三、核心子系统

3.1 AIAgent 核心引擎

文件: run_agent.py (~9,200 行)

核心职责:
- 组装有效系统提示词和工具架构
- 选择正确的 Provider/API 模式
- 执行可中断的模型调用
- 执行工具调用（顺序或并发）
- 维护对话历史（OpenAI 消息格式）
- 处理压缩、重试、回退模型切换
- 跟踪父代理和子代理的迭代预算
- 在上下文丢失前刷新持久记忆

API 模式

模式 用途 客户端类型
chat_completions OpenAI 兼容端点 openai.OpenAI
codex_responses OpenAI Codex/Responses API openai.OpenAI
anthropic_messages Anthropic 原生 API anthropic.Anthropic

模式解析顺序:
1. 显式 api_mode 构造函数参数（最高优先级）
2. Provider 特定检测（如 anthropic → anthropic_messages）
3. Base URL 启发式（如 api.anthropic.com → anthropic_messages）
4. 默认: chat_completions

轮次生命周期

run_conversation()
  1. 生成 task_id（如未提供）
  2. 追ser 用户消息到对话历史
  3. 构建或重用缓存的系统提示词
  4. 检查是否需要预压缩（>50% 上下文）
  5. 从对话历史构建 API 消息
  6. 注入临时提示词层（预算警告、上下文压力）
  7. 应用提示词缓存标记（如在 Anthropic）
  8. 执行可中断 API 调用
  9. 解析响应:
     - 如 tool_calls: 执行它们，追加结果，循环回步骤 5
     - 如文本响应: 持久化会话，如需要刷新记忆，返回

消息格式

所有消息使用 OpenAI 兼容格式：

{"role": "system", "content": "..."}
{"role": "user", "content": "..."}
{"role": "assistant", "content": "...", "tool_calls": [...]}
{"role": "tool", "tool_call_id": "...", "content": "..."}

推理内容: 存储在 assistant_msg["reasoning"]，可选通过 reasoning_callback 显示。

消息交替规则

Agent 循环强制严格的消息角色交替：
- 系统消息后: User → Assistant → User → Assistant → ...
- 工具调用期间: Assistant (with tool_calls) → Tool → Tool → ... → Assistant
- 永不两个 assistant 消息连续
- 永不两个 user 消息连续
- 只有 tool 角色可以有连续条目（并行工具结果）

可中断 API 调用

┌──────────────────────┐ ┌──────────────┐
│ Main thread         │ │ API thread    │
│ wait on:            │────▶│ HTTP POST    │
│ - response ready     │ │ to provider   │
│ - interrupt event    │ └──────────────┘
│ - timeout           │
└──────────────────────┘

当被中断（用户发送新消息、/stop 命令或信号）：
- API 线程被放弃（响应丢弃）
- Agent 可以处理新输入或干净关闭
- 不会将部分响应注入对话历史

工具执行

顺序 vs 并发:
- 单个工具调用 → 直接在主线程执行
- 多个工具调用 → 通过通过 ThreadPoolExecutor 并发执行

例外：标记为交互式的工具（如 clarify）强制顺序执行。

执行流程:

for each tool_call in response.tool_calls:
  1. 从 tools/registry.py 解析处理器
  2. 触发 pre_tool_call 插件 hook
  3. 检查是否危验命令 (tools/approval.py)
     - 如危验: 调用 approval_callback，等待用户
  4. 用 args + task_id 执行处理器
  5. 触发 post_tool_call 插件 hook
  6. 追ser {"role": "tool", "content": result} 到历史

Agent 级工具

有些工具在到达 handle_function_call() 前被 run_agent.py 拦截：

工具 拦截原因
todo 读写 agent 本地任务状态
memory 写入持久记忆文件
session_search 查询 agent 的会话 DB
delegate_task 生成子代理会话

这些工具直接修改 agent 状态并返回合成工具结果，不通过注册表。

回调表面

回调 触发时机 使用者
tool_progress_callback 工具执行前后 CLI spinner、网关进度消息
thinking_callback 模型开始/停止思考 CLI "thinking..." 指示器
reasoning_callback 模型返回推理内容 CLI 推理显示、网关推理块
clarify_callback clarify 工具被调用 CLI 输入提示、网关交互消息
step_callback 每个完整 agent 轮次后 网关步骤跟踪、ACP 进度
stream_delta_callback 每个流式 token（启用时） CLI 流式显示
tool_gen_callback 从流解析工具调用 CLI spinner 中工具预览
status_callback 状态变化 ACP 状态更新

预算和回退行为

迭代预算:
- 默认: 90 轮次（可通过 agent.max_turns 配置）
- 在父代理和子代理间共享
- 两层预算压力:
- 70%+ 使用: 追ser [BUDGET: Iteration X/Y. N iterations left. Start consolidating your work.]
- 90%+ 使用: 追ser [BUDGET WARNING: Iteration X/Y. Only N iteration(s) left. Provide your final response NOW.]
- 100%: Agent 停止并返回工作摘要

回退模型:
当主模型失败（429 速率限制、5xx 服务器错误、401/403 认证错误）：
- 检查配置中的 fallback_providers 列表
- 按顺序尝试每个回退
- 成功后用新 provider 继续对话
- 401/403 时，失败前尝试凭证刷新

  ↓
model_tools.handle_function_call(name, args, task_id, user_task)
  ↓
[Agent-loop tools?] → 直接由 agent 循环处理

[Plugin pre-hook] → invoke_hook("pre_tool_call", ...)

  ↓

  ↓

[Sync handler?] → 直接调用
  ↓

  ↓

四、长期记忆系统

4.1 内置记忆（核心层）

Hermes 采用双层记忆架构，在会话开始时注入系统提示词：

记忆文件 用途 字符限制 Token 估算
MEMORY.md Agent 个人笔记（环境事实、约定、教训） 2,200 字符 ~800 tokens
USER.md 用户画像（偏好、沟通风格、期望） 1,375 字符 ~500 tokens

特点:
- 冻结快照模式: 在会话开始时一次性注入，会话中不改变
- Agent 自主管理: 通过 memory 工具自动 add/replace/remove
- 字符限制: 保持记忆聚焦，强制精集
- 安全扫描: 检测提示注入、凭证泄露、SSH 后门等威胁

记忆注入格式:

══════════════════════════════════════════════
MEMORY (your personal notes) [67% — 1,474/2,200 chars]
══════════════════════════════════════════════
User's project is a Rust web service at ~/code/myapi using Axum + SQLx
§
This machine runs Ubuntu 22.04, has Docker and Podman installed
§
User prefers concise responses, dislikes verbose explanations

记忆工具操作:
- add - 添加新记忆条目
- replace - 替换现有条目（使用子串匹配）
- remove - 移除不再相关的条目（使用子串匹配）

六、上下文压缩与缓存

6.1 双层压缩系统

┌──────────────────────────┐
│ Gateway Session Hygiene │ 85% 阈值
│ (预 Agent，粗略估算)    │ 安全网
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ Agent ContextCompressor  │ 50% 阈值（可配置）
│ (循环内，真实 Token)     │ 主要管理
└──────────────────────────┘

配置参数:

compression:
  enabled: true
  threshold: 0.50        # 50% 上下文窗口时触发
  target_ratio: 0.20     # 尾部保护 20% 阈值预算
  protect_last_n: 20       # 最少保护 20 条消息
  summary_model: null      # 摘要模型（默认 auxiliary）
）

参数详情:

参数 默认值 范围 说明
threshold 0.50 0.0-1.0 当 prompt tokens ≥ threshold × context_length 时触发压缩
target_ratio 0.20 0.10-0.80 控制尾部保护 token 预算: threshold_tokens × target_ratio
protect_last_n 20 ≥1 始终保留的最少消息数
protect_first_n 3 (硬编码) 始终保护系统提示词 + 首次交换

计算值（200K 上下文模型，默认配置）:

context_length = 200,000
threshold_tokens = 200,000 × 0.50 = 100,000
tail_token_budget = 100,000 × 0.20 = 20,000
max_summary_tokens = min(200,000 × 0.05, 12,000) = 10,000

6.2 压缩算法（四阶段）

Phase 1: 修剪旧工具结果（无 LLM 调用）

受保护尾部之外的旧工具结果（>200 字符）被替换为：

[Old tool output cleared to save context space]

这是一个廉价的预通过，从冗长的工具输出（文件内容、终端输出、搜索结果）中节省大量 token。

Phase 2: 确定边界

┌─────────────────────────────────────────────────────────────┐
│ Message list                                             │
│                                                         │
│ [0..2] ← protect_first_n (系统提示词 + 首次交换)       │
│ [3..N] ← middle turns → 摘要化                         │
│ [N..end] ← tail (by token budget OR protect_last_n)        │
│                                                         │
└─────────────────────────────────────────────────────────────┘

尾部保护基于 token 预算：从末尾向后走，累积 token 直到预算耗尽。如果预算保护的消息数少于 protect_last_n，则回退到固定计数。

边界对齐以避免分割 tool_call/tool_result 组。_align_boundary_backward() 方法走过连续的 tool 结果以找到父 assistant 消息，保持组完整。

Phase 3: 生成结构化摘要

中间轮次使用辅助 LLM 和结构化模板进行摘要：

## Goal
[用户试图完成什么]

## Constraints & Preferences
[用户偏好、编码风格、约束、重要决策]

## Progress
### Done
[已完成工作 - 具体文件路径、命令、结果]
### In Progress
[当前进行的工作]
### Blocked
[阻塞或遇到的问题]

## Key Decisions
[重要技术决策及原因]

## Relevant Files
[读取、修改或创建的文件 - 简短说明]

## Next Steps
[接下来需要做什么]

## Critical Context
[具体值、错误消息、配置详情]

摘要预算根据压缩的内容量缩放：
- 公式: content_tokens × 0.20（_SUMMARY_RATIO 常量）
- 最小: 2,000 tokens
- 最大: min(context_length × 0.05, 12,000) tokens

Phase 4: 组装压缩消息

压缩消息列表为：
- 头部消息（首次压缩时在系统提示词附加注释）
- 摘要消息（角色选择以避免连续同角色违规）
- 尾部消息（未修改）

孤立的 tool_call/tool_result 对通过 _sanitize_tool_pairs() 清理：
- 引用已移除调用的工具结果 → 移除
- 结果被移除的工具调用 → 注入存根结果

6.3 迭代式重新压缩

在后续压缩中，前摘要传递给 LLM 并指示更新而非从头摘要。这跨多次压缩保留信息——项目从 "In Progress" 移到 "Done"，添加新进度，移除过时信息。

压缩器实例上的 _previous_summary 字段为此目的存储上次摘要文本。

6.5 Anthropic 提示词缓存

策略: system_and_3

Anthropic 允许每个请求最多 4 个 cache_control 断点。Hermes 使用 "system_and_3" 策略：

Breakpoint 1: 系统提示词（所有轮次稳定）
Breakpoint 2: 倒数第 3 条非系统消息 ─┐
Breakpoint 3: 倒数第 2 条非系统消息 ├─ 滚动窗口
Breakpoint 4: 最后一条非系统消息 ─┘

工作原理:

apply_anthropic_cache_control() 深拷消息并注入缓存标记：

# 缓存标记格式
marker = {"type": "ephemeral"}
# 或 1 小时 TTL:
marker = {"type": "ephemeral", "ttl": "1h"}

标记根据内容类型不同应用：

内容类型 标记位置
String content 转换为 [{"type": "text", "text": ..., "cache_control": ...}]
List content 添加到最后元素的 dict
None/empty 添加为 msg["cache_control"]
Tool messages 添加为 msg["cache_control"]（仅原生 Anthropic）

缓存感知设计模式:
- ✅ 稳定系统提示词：系统提示词是断点 1 并跨所有轮次缓存。避免对话中途变异（压缩仅在首次压缩时附加注释）
- ✅ 消息顺序很重要：缓存命中需要前缀匹配。在中间添加或移除消息会使之后的所有内容缓存失效
- ✅ 压缩缓存交互：压缩后，压缩区域的缓存失效，但系统提示词缓存存活。滚动的 3 消息窗口在 1-2 轮内重建缓存
- ✅ TTL 选择：默认 5m（5 分钟）。对长运行会话使用 1h，其中用户在轮次之间休息

启用提示词缓存:

提示词缓存在以下情况自动启用：
- 模型是 Anthropic Claude 模型（通过模型名检测）
- Provider 支持 cache_control（原生 Anthropic API 或 OpenRouter）

# config.yaml — TTL 可配置
model:
  cache_ttl: "5m"  # "5m" 或 "1h"

CLI 在启动时显示缓存状态：

💾 Prompt caching: ENABLED (Claude via OpenRouter, 5m TTL)

6.6 上下文压力警告

Agent 在压缩阈值的 85%（不是 85% 上下文——85% 的阈值为 50% 上下文）时发出上下文压力警告：

⚠️ Context is 85% to compaction threshold (42,500/50,000 tokens)

压缩后，如果使用率降到阈值的 85% 以下，警告状态被清除。如果压缩未能将使用率降到警告水平以下（对话太密集），警告持续，但压缩不会在再次超过阈值前重新触发。

- 复杂任务后自动提取工作流
- 错误和纠正驱动技能改进
- 技能通过 Skills Hub 共享和复用

3. 上下文智能压缩
- 结构化摘要保留关键信息
- 迭代式更新避免信息丢失
- Anthropic 缓存降低成本

4. 跨会话知识检索
- FTS5 全文搜索快速定位
- LLM 摘要提取关键信息
- 外部 Provider 提供语义搜索

5. 多层记忆架构
- 内置记忆提供即时访问
- 外部 Provider 提供深度知识
- 会话搜索连接历史对话

八、设计原则

原则 实践
提示词稳定性 系统提示词不会在对话中途改变。除非显式用户操作（/model），否则无缓存破坏变异
可观察执行 每个工具调用对用户可见（通过回调）。CLI 中的进度更新（spinner）和网关中的聊天消息
可中断 API 调用和工具执行可中途取消（通过用户输入或信号）
平台无关核心 一个 AIAgent 类服务 CLI、网关、ACP、批量和 API 服务器。平台差异存在于入口点，而非 agent
松耦合 可选子系统（MCP、插件、memory providers、RL 环境）使用注册模式和 check_fn 门控，而非硬依赖
Profile 配置隔离 每个 profile（hermes -p <name>）获得自己的 HERMES_HOME、配置、记忆、会话和网关 PID。多个 profile 可并发运行

九、数据流详解

9.1 CLI 会话流程

用户输入
  → HermesCLI.process_input()
  → AIAgent.run_conversation()
  → prompt_builder.build_system_prompt()
  → runtime_provider.resolve_runtime_provider()
  → API 调用
  → tool_calls? → model_tools.handle_function_call()
  → 循环
  → 最终响应 → 显示 → 保存到 SessionDB

9.2 网关消息流程

平台事件
  → Adapter.on_message()
  → MessageEvent
  → GatewayRunner._handle_message()
  → 授权用户
  → 解析会话 key
  → 创建 AIAgent + 会话历史
  → AIAgent.run_conversation()
  → 通过适配器投递响应

9.3 Cron 任务流程

调度器 tick
  → 从 jobs.json 加载到期任务
  → 创建新的 AIAgent（无历史）
  → 注入附加技能作为上下文
  → 运行任务提示词
  → 投递到目标平台
  → 更新任务状态和 next_run

十、安全特性

10.1 危险命令检测

检测模式 (tools/approval.py):

类别 模式示例
递归删除 rm -rf
文件系统格式化 mkfs, dd
SQL 破坏性操作 DROP TABLE, DELETE FROM without WHERE
系统配置覆盖 > /etc/
服务操作 systemctl stop
远程代码执行 curl \| sh
Fork bombs, 进程 kills :(){ :|:& };:, kill -9

10.2 命令批准流程

  1. 模式检测: detect_dangerous_command(command) 检查所有模式
  2. 批准提示: 如匹配：
    - CLI 模式: 交互式提示要求用户批准、拒绝或永久允许
    - 网关模式: 异步批准回调发送请求到消息平台
  3. 智能批准: 可选地，辅助 LLM 可自动批准匹配模式的低风险命令（如 rm -rf node_modules/ 安全但匹配 "recursive delete"）
  4. 会话状态: 批准按会话跟踪。一旦你为会话批准 "recursive delete"，后续 rm -rf 命令不再提示
  5. 永久 allowlist: "永久允许" 选项将模式写入 config.yaml 的 command_allowlist，跨会话持久

10.3 记忆安全扫描

记忆条目在注入系统提示词前被扫描，因为它们被注入到系统提示词中。匹配威胁模式（提示注入、凭证泄露、SSH 后门）或包含不可见 Unicode 字符的内容被阻止。

十三、总结

Hermes Agent 是一个设计精良、功能完备的 AI 助手框架，特别适合需要长期记忆、多平台集成和工具调用能力的场景。

核心优势

  1. 真正的自演进能力: 通过技能系统和记忆管理，Agent 能够从经验中学习并持续改进
  2. 多层记忆架构: 内置记忆 + 8 个外部 Provider，提供从即时访问到深度知识图谱的完整谱系
  3. 智能上下文管理: 双层压缩 + Anthropic 缓存，优化 Token 使用和成本
  4. 平台无关核心: 单一 AIAgent 类服务所有入口点，易于扩展和维护
  5. 安全优先: 危险命令检测、批准流程、记忆安全扫描等多层防护
  6. 灵活部署: 从 $5 VPS 到 GPU 集群，适应各种使用场景

适用场景

  • 需要长期记忆和跨会话上下文的个人助手
  • 多平台消息集成（Telegram、Discord、Slack 等）
  • 自动化任务调度和定时执行
  • 研究和开发工作流自动化
  • 需要从经验中学习并持续改进的 AI 系统

设计哲学

Hermes Agent 的自演进机制体现了几个核心设计原则：

  1. 渐进式披露 - 只在需要时加载完整内容
  2. 可观察执行 - 所有工具调用对用户可见
  3. 提示词稳定性 - 系统提示词不会中途改变
  4. 松耦合 - 可选子系统使用注册模式
  5. Token 效率 - 缓存、压缩、分层加载
  6. 安全优先 - 扫描、批准、隔离

这是一个真正能够从经验中学习的 AI 助手框架，随着使用时间增长，它会变得更了解用户、更高效、更智能。
